# 正确性

当我们编写代码时，我们总是希望自己能写对。我们*希望*我们的代码
是正确的。但我们怎么才能“知道”它是正确的呢？在本章中，我们将学习
三个可能的答案：文档、测试和证明。

说实话：我们都曾一度觉得文档或测试无聊、乏味，而且完全可以推迟。
但随着程序员逐渐成熟，他们会意识到，这两者对于写出正确代码都至关重要。
它们分别记录了关于代码的两类“真相”。

**文档**记录的是程序员的意图，而不是程序实际做了什么。
它把作者脑中的想法传达给其他人。很多时候（即使在本书中！），
我们都没能按预期表达清楚自己的想法。问题可能出在代码里，
也可能出在文档里。但编写文档会迫使我们再次（嗯，*希望*是再次）
思考自己的意图。向人解释想法和向计算机解释想法，是两种不同的认知任务；
这个过程会暴露我们思维中的漏洞。

更重要的是，文档是时间胶囊里的消息。想象一下：
某个很遥远、现在已经联系不上的人，把这条消息发给了你这个程序员。
你需要它来解释眼前的“考古证据”&mdash;也就是你继承下来的、
难以理解的源代码。你唯一的希望是，原作者当年足够有同理心，
把自己的想法写了下来。

现在再想象一下：那个来自遥远过去的作者，**如果就是你自己呢？**
也许是两周前、两个月前，或者两年前的你。人的记忆很短暂。
如果你才刚编程几年，这一点可能还不明显，但请相信：
总有一天你会回到今天写的代码面前，却完全不记得它是什么意思。
你唯一的希望，就是在写下它的时候给未来的自己留下一些线索；
否则再次回来时，你会迷路。

**测试**记录的是程序实际做了什么，而不是程序员原本想让它做什么。
它提供证据，说明程序员的理解是否正确。优秀的科学家需要证据。
这种要求不是出于傲慢，而是出于谦逊：人类非常擅长欺骗自己。
（考虑一下现代社交媒体的回声室。）你可以写一段
你*认为*正确的代码。但你可以写一个测试用例，为它的正确性提供证据。
然后再写十个。证据不断积累，最终就足以令人信服。
这是绝对的吗？当然不是。也许还有一些测试用例是你没有想到的。
这就是科学：新想法的出现挑战旧想法。

更重要的是，测试是一种“可重复”的科学。实验能否被重复，
对它所支撑的结论至关重要。把测试写成单元测试套件，
就是把实验变成可以自动重复运行的形式。这样，我们就能向现在和未来的
自己及他人展示：代码在这些情况下是正确的。此外，当代码随时间被修改时，
测试还能帮助我们保持这种正确性，无论修改者是别人还是我们自己。

**文档和测试的挑战**在于纪律。只关心写代码实在太诱人：
“这才是有趣的部分”，对吧？但这就像一封信只写了三分之一。
一部分是写给机器的，说明如何计算；另一部分是写给其他人的，
说明我们想计算什么；还有一部分同时写给机器和人，
说明我们实际能计算什么。只有这三部分都写完，工作才算完成。

如果你还不确信文档和测试的重要性，
不用担心。只要你长期打磨这门手艺，未来自然会被说服。
与此同时，让我们先学习如何把它们做得更好。本章会介绍一些有效的
（也希望对你来说是新的）文档和测试技术。

最后，除了文档和测试之外，还有数学**证明**
正确性。逻辑和离散数学技术可用于形式化地证明程序满足规格说明。
这样的证明不一定容易&mdash;事实上，它们比文档和测试需要更多训练和纪律。
但当程序用于可能危及生命的安全关键任务时，使用这些证明就很有意义。
