# 测试与调试

正确的程序会按照我们的预期运行。*验证*是一个过程
建立我们对正确的程序行为的信心。

## 验证

有很多方法可以增强这种信心。社会方法、形式化方法和测试是其中三类。
测试是我们的主要关注点，但我们先考虑另外两类。

**社会方法**涉及与其他人一起开发程序，借助他们的帮助提高正确性。
一些好的技术包括：

- *代码走查。* 在走查方法中，程序员向审查团队展示文档和代码，
  然后团队给出反馈。这是一个非正式过程。关注点是代码而不是编码者，
  因而更容易避免伤害感情。但团队对代码正确性的信心可能不会提高太多。

- *代码检查。* 在这种方法中，审查团队主导代码审查过程。
  团队提前做一些准备是有用的，尽管不必准备太多。
  他们会定义审查过程的目标，并与编码者互动，了解哪里可能存在质量问题。
  同样，让这个过程尽量不带指责很重要。

- *结对编程。* 最非正式的代码审查方法是结对编程：
  代码由两名工程师共同开发，一人作为驾驶员编写代码，另一人作为观察者观看。
  观察者的职责是提出批评、思考潜在错误，并帮助把握更大的设计问题。
  通常最好让对当前编码任务更有经验的工程师担任观察者。
  观察者审查代码，扮演驾驶员必须说服的“反方”。当两人在制定规格说明时，
  观察者会思考如何让规格说明更清晰或更简短。结对编程还有其他好处：
  和伙伴一起工作往往更有趣、更有教育意义，也有助于双方专注于任务。
  如果你刚开始和另一位程序员合作，结对编程是了解对方思路并建立共同词汇的好方法。
  伙伴之间也最好轮换角色。

这些用于“代码审查”的社会技术非常有效。在 IBM 的一项研究中
（Jones，1991），代码检查发现了 65% 的已知编码错误和 25% 的已知文档错误；
而测试只发现了 20% 的编码错误，并且没有发现任何文档错误。
代码检查过程可能比走查更有效。一项研究（Fagan，1976）发现，
与代码走查相比，代码检查让代码故障减少了 38%。

然而，彻底的代码审查可能很昂贵。Jones 发现，
每 150 行代码的检查准备需要 1 小时，而实际检查每小时覆盖 75 行代码。
最多三人的检查团队能提高检查质量；超过三人后，更多检查者似乎没有帮助。
花大量时间准备检查似乎也没有帮助。也许这是因为检查的大部分价值来自
与编码者的互动。

**形式化方法**利用数学和逻辑的力量来验证程序行为。*验证*使用程序代码及其规格说明来
构造一个证明，证明程序在所有可能的输入上都能正确运行。
通常有一些研究工具可以帮助程序验证，包括基于自动定理证明器的工具，
以及专为程序验证设计的研究语言。验证往往很昂贵，并且需要仔细思考、
深入理解待验证代码。所以在实践中，它往往适用于重要且相对较短的代码。
验证对于测试效果较差的关键系统特别有价值。因为并发程序的执行不是确定性的，
它们很难测试；有时，细微错误只能通过尝试形式化验证代码来发现。
事实上，辅助证明程序正确性的工具正变得越来越有效，一些大型系统已经得到全面验证，
包括编译器、处理器、处理器模拟器，以及操作系统的关键部分。

**测试**涉及在示例输入上实际执行程序以查看
行为是否符合预期。通过对比实际结果
程序达到预期结果，我们看看程序是否真的有效
在我们尝试的特定输入上。测试永远不能提供绝对的
保证形式化方法所能提供的绝对性，但它更容易、更便宜。
它也可能是你最熟悉的验证方法。测试是一种良好且经济高效的方式，
用于建立我们对程序正确行为的信心。

## 调试

当测试发现错误时，我们通常说这个程序 “buggy”。但是
“bug”这个词听起来像是程序里凭空出现了某种东西。更好的术语是：

* *故障*，也就是人为错误在软件系统中留下的结果；以及

* *失败*，即违反要求。

有些故障可能永远不会暴露给系统的最终用户，但失败则是那些确实暴露出来的错误。
故障可能是因为实现没有匹配设计，也可能是因为设计本身不符合需求。

*调试*是发现并修复故障的过程。测试显然属于“发现”部分，
但修复可能更复杂。调试花费的时间可能比最初实现本身还多。
所以你应该从一开始就让程序容易调试：为每个函数写好规格说明，
记录每个数据抽象的 AF 和 RI，保持模块较小，并独立测试它们。

但不可避免的是，你会发现程序中的错误。当你这样做时，
通过采用*科学方法*像科学家一样处理它们：

* 评估可用数据；

* 提出一个可以解释数据的假设；

* 设计一个可重复的实验来检验该假设；和

* 使用该实验的结果来完善或反驳你的假设。

这个过程的关键通常是找到能触发故障的最简单、最小输入。
它通常不是我们最初发现故障时使用的输入。因此，可能需要一些初步实验
来找到*最小测试用例*。

不要害怕编写额外代码，甚至很多额外代码，来帮助你发现故障。
像 `to_string` 或 `format` 这样的函数对于理解计算很有帮助，
因此即使在检测到任何错误之前就写好它们，也是完全值得的。

当你确实找到故障根源时，修复时要格外小心。快速改一下代码然后继续前进
很诱人，但这相当危险。很多时候，修复一个故障只会引入新的、
未知的故障。如果一个 bug 很难发现，往往是因为程序逻辑复杂、难以推理。
请仔细思考这个错误最初为什么会被引入，以及未来如何防止类似故障。
