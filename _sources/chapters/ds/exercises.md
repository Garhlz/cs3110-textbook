{{ ex4 | replace("%%NAME%%", "linear probing")}}

我们简要提到*探测*作为*链接*的替代方案。探测可以是
有效地用于哈希表的硬件实现以及
数据库。通过探测，每个桶只包含一个绑定。如果出现
冲突时，我们通过数组向前搜索，如下所述。

**你的任务：** 实现一个使用线性探测的哈希表。详细信息是
下面。

**查找。** 假设我们试图在表中查找绑定。我们散列
绑定的密钥并在适当的存储桶中查找。如果已经有一个
该存储桶中的不同键，我们开始在数组中向前搜索
下一个桶，然后是下一个桶，依此类推，绕回到
如有必要，位于数组的开头。最终我们要么

* 找到一个空桶，在这种情况下我们要搜索的键没有绑定
在表中；

* 在到达空桶之前找到密钥，在这种情况下我们可以返回
价值；或

* 永远找不到键或空桶，而是绕回原处
原始桶，这种情况下所有桶都已满并且密钥未绑定
  在表中。这种情况实际上不应该发生，因为我们不允许
  负载系数足够高以填充所有桶。

**插入。** 插入遵循与查找键相同的算法，不同之处在于
每当我们第一次找到一个空桶时，我们就可以在那里插入绑定。

**移除。** 移除更加困难。一旦找到键，我们就不能
桶是空的，因为这会导致未来的搜索受到影响
早点停下来。相反，我们可以在该存储桶中引入一个特殊的“已删除”值
指示存储桶不包含绑定，但搜索应该
不止于此。

**调整大小。** 由于我们不希望数组完全填满，因此我们可以
保持负载系数接近 1/4。当负载系数超过1/2时，我们可以加倍
阵列，使负载系数回到 1/4。当负载系数低于
1/8，我们可以将阵列减半，再次将负载系数恢复到 1/4。
“删除”绑定使负载因子的定义变得复杂：

* 当确定是否将表大小加倍时，我们计算负载
系数为（绑定数量 + 已删除绑定数量）/（存储桶数量）。也就是说，
  删除的绑定有助于增加负载因子。

* 当确定是否是表大小的一半时，我们计算负载因子
为（绑定数量）/（存储桶数量）。也就是说，删除的绑定不算
  从而提高负载率。

重新散列表时，删除的绑定当然不会重新插入到
新表。

{{ ex3 | replace("%%NAME%%", "efficient traversal")}}

假设你想将树转换为列表。你必须输入值
以某种顺序存储在树中。以下是三种方法：

* *preorder*：每个节点的值出现在列表中其值之前
左子树然后右子树。

* *中序*：出现左子树的值，然后是节点处的值，
然后是右子树的值。

* *postorder*：节点的左子树和右子树的值出现，紧随其后
通过节点处的值。

这是实现这些*遍历*的代码，以及一些示例
应用：

```ocaml
type 'a tree = Leaf | Node of 'a tree * 'a * 'a tree

let rec preorder = function
  | Leaf -> []
  | Node (l,v,r) -> [v] @ preorder l @ preorder r

let rec inorder = function
  | Leaf -> []
  | Node (l,v,r) ->  inorder l @ [v] @ inorder r

let rec postorder = function
  | Leaf -> []
  | Node (l,v,r) ->  postorder l @ postorder r @ [v]

let t =
  Node(Node(Node(Leaf, 1, Leaf), 2, Node(Leaf, 3, Leaf)),
       4,
       Node(Node(Leaf, 5, Leaf), 6, Node(Leaf, 7, Leaf)))

(*
  t is
        4
      /   \
     2     6
    / \   / \
   1   3 5   7
*)

let () = assert (preorder t  = [4;2;1;3;6;5;7])
let () = assert (inorder t   = [1;2;3;4;5;6;7])
let () = assert (postorder t = [1;3;2;5;7;6;4])
```

在不平衡树上，上面的遍历函数需要二次最坏情况
时间（以节点数计），因为 `@` 运算符。重新实施
没有 `@` 的函数，而是使用 `::`，这样它们就能准确执行
树中的每个 `Node` 都有一次 cons。因此，最坏情况的执行时间将是
线性。你需要为每个添加一个额外的累加器参数
函数，很像尾递归。 （但是你的实现不会
实际上是尾递归。）

{{ ex4 | replace("%%NAME%%", "approximately e")}}

指数函数 $e^x$ 可以通过以下方式计算
无穷总和：

$$
e^x = \frac{x^0}{0!} + \frac{x^1}{1!} + \frac{x^2}{2!} + \frac{x^3}{3!} + \cdots + \frac{x^k}{k!} + \cdots
$$

定义一个函数 `e_terms : float -> float sequence`。元素 `k`
序列应该是无限和中的项 `k` 。例如，`e_terms 1.0` 是
序列 `<1.0; 1.0; 0.5; 0.1666...; 0.041666...; ...>`。简单的方法
计算涉及计算 $f(k) = \frac{x^k}{k!}$ 的函数。

定义一个函数 `total : float sequence -> float sequence`，使得
`total <a; b; c; ...>` 是输入元素的运行总计，即
`<a; a+.b; a+.b+.c; ...>`。

通过一起使用 `e_terms` 和 `total` ，你将能够计算连续的
$e^x$ 的近似值对应于无限的有限前缀
总结。例如，你可以计算流
`<1.; 2.; 2.5; 2.66666666666666652; 2.70833333333333304; ...>`。它包含
$e^1$ 的连续近似，使得流的元素 $n$ 是
$\sum_{k=0}^{n} \frac{1^k}{k!}$。

定义一个函数 `within : float -> float sequence -> float`，使得
`within eps s` 是 `s` 的第一个元素，其绝对差值
该元素与其之前的元素之间的距离严格小于 `eps`。如果
不存在这样的元素，允许 `within` 不终止（即进入
“无限循环”）。作为先决条件，*容差* `eps` 必须严格遵守
积极的。例如，
`within 0.1 <1.0; 2.0; 2.5; 2.75; 2.875; 2.9375; 2.96875; ...>` 是 `2.9375`。

最后，定义一个函数 `e : float -> float -> float` ，使得 `e x eps` 为
$e^x$ 使用上述无限求和的有限前缀进行计算。
当连续值之间的绝对差值达到时，计算应停止
近似值低于 `eps`，它必须严格为正数。对于
例如，`e 1. 0.01` 将是 `2.71666666666666634`。
