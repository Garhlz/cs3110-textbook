# 模块系统

编程语言的*模块系统*，是该语言提供的一组用于支持模块化编程的机制。
下面列出模块系统通常需要处理的几个问题。讨论时我们主要比较 Java 和
OCaml，并指出两种语言中最相关的机制。

**命名空间。** *命名空间*提供一组组合在一起的名称，
这些名称通常在逻辑上相关，并且与其他命名空间隔离。
因此，一个命名空间中的名称 `foo` 可以和另一个命名空间中的 `foo`
有不同含义。命名空间是一种作用域机制，对模块化至关重要。
没有命名空间，一个程序员选择的名称很容易与另一个程序员选择的名称冲突。
在 Java 中，类和包会把名称组织起来。在 OCaml 中，
*结构*（我们很快就会学习）也类似地把名称组织在一起，但它们不带有
类在面向对象编程中通常伴随的复杂性
（构造函数、静态成员与实例成员、继承、重写、`this`、
等）。结构是 OCaml 模块系统的核心；事实上，我们一直在使用它们，
只是还没有仔细讨论。

**抽象。** *抽象*隐藏一些信息，同时揭示其他信息
信息。因此，抽象可以实现*封装*，也就是*信息隐藏*。
通常，模块的抽象机制允许暴露模块内部的一些名称，同时隐藏另一些名称。
因此，抽象描述的是模块之间的一种关系：许多不同模块都可以被看作满足
同一个抽象。抽象对于模块化至关重要，因为它让模块实现者能够向客户端
隐藏实现细节，从而防止客户端依赖或滥用这些细节。在大型团队中，
这可以保护一个程序员设计的模块不被另一个程序员误用；同时也让客户端
不必知道这些细节。因此，在大型团队中，没有任何程序员需要了解所有模块的
全部细节。在 Java 中，接口和抽象类提供了抽象。在 OCaml 中，
*签名* 通过隐藏结构中的某些名称和定义来抽象结构。签名本质上就是结构的类型。

**代码重用。** 模块系统还会提供*代码重用*机制，使一个模块中的代码
可以成为另一个模块的一部分，而不必手动复制。代码重用让程序员能够以
可维护的方式建立在其他模块之上：当某个模块的实现者改进了该模块，
所有重用这段代码的程序员都会自动受益。代码重用对于模块化很关键，
因为它让我们能够把“构建块”反复组装成复杂的软件。在 Java 中，
子类型和继承提供代码重用。在 OCaml 中，*函子* 和 *include*
提供代码重用。函子像函数一样，从已有模块生成新模块。`include`
则像一种有原则的复制粘贴：把程序某个部分的代码纳入另一个部分。

```{warning}
Java 和 OCaml 之间的这些类比必然不完美。读完上面的讨论后，
你可能会很自然地想到下面任一种说法：

- “结构就像 Java 类，签名就像接口。”

- “结构就像 Java 对象，签名就像类。”

两种说法都有一定帮助，但最终都不准确。所以此时最好暂时放下
面向对象编程的类比，直接理解 OCaml 模块系统本身。它和 Java
的组织方式确实不同。
```
