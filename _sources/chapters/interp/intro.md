熟练的工匠必须了解自己工作时使用的工具。木匠需要了解锯子和刨子。
厨师需要了解刀具和锅具。程序员则需要了解编译器等工具，
以及他们所使用的编程语言是如何实现的。

充分理解编译需要一两门完整的课程。所以在这里，
我们将简要了解如何实现编程语言。目标是掌握一些基本实现技术，
从而揭开你正在使用的工具的神秘面纱。虽然你可能永远不需要
实现一门完整的通用编程语言，但在职业生涯的某个时刻，
你很可能会想要设计并实现一些小型的、特殊用途的语言。有时这些被称为
*特定领域语言*（DSL）。我们在这里介绍的内容应该可以帮助你
任务。

*编译器*是实现编程语言的程序。*解释器*也是如此。
但二者的实现策略不同。

编译器的主要任务是*翻译*。它以*源程序*作为输入
并生成*目标程序*作为输出。源程序通常是
用高级语言（例如 Java 或 OCaml）表达。目标程序是
通常用低级语言表示，例如 MIPS 或 x86 汇编。然后
编译器的工作已经完成，之后不再需要它；随后由操作系统帮助加载并执行目标程序。
通常，编译得到的实现性能更高。

解释器的主要任务是*执行*。它以源程序作为输入
并直接执行该程序，而不产生任何目标程序。操作系统实际加载并执行的是解释器，
然后由解释器负责执行源程序。通常情况下，解释器比编译器更容易实现。

也可以混合使用编译和解释来实现一门语言。
最常见的例子是由“虚拟机”执行*字节码*，
例如 Java 虚拟机 (JVM) 或 OCaml 虚拟机（以前称为 Zinc Machine）。
采用这种策略时，
编译器将源语言翻译成字节码，虚拟机
解释字节码。

高性能虚拟机（例如 Java 的 HotSpot）会更进一步：
在虚拟机中嵌入编译器。当虚拟机注意到某段字节码被频繁解释执行时，
它会使用编译器把这段字节码翻译成当前机器上的机器语言（例如 x86）。
这称为*即时编译* (JIT)，因为代码是在即将执行时才被编译的。

编译器在翻译程序时会经历几个阶段：

**词法分析。** 在词法分析过程中，编译器会把程序的原始源代码
从字符序列转换为*标记*序列。标记是组合在一起后具有某种含义的相邻字符。
你可以将它们视为自然语言中的单词。确实，诸如此类的关键词
比如 `if` 和 `match` 会是 OCaml 中的标记。诸如 `42` 之类的常量也是如此
和 `"hello"`，变量名称，例如 `x` 和 `lst`，以及标点符号，例如
`(`、`)` 和 `->`。词法分析通常会删除空格，因为它不再是
一旦识别出标记就不再需要。（尽管在空白敏感的
像 Python 这样的语言，则需要保留它。）

**解析。** 在解析过程中，编译器将标记序列转换为
一棵称为*抽象语法树*（AST）的树。顾名思义，这棵树
从语言的“具体语法”中抽象出来。回想一下那个抽象
可能意味着“忘记细节”。AST 通常会丢弃具体语法中的许多细节。例如：

* 在 `1 + (2 + 3)` 中，括号对右侧加法运算进行分组，
表明应首先对其进行评估。一棵树可以表示如下：
  ```
     +
    / \
   1   +
      / \
     2   3
  ```
不再需要括号，因为树的结构已经编码了分组信息。

* 在 `[1; 2; 3]` 中，方括号描述了
列表，分号分隔列表元素。一棵树可以代表
  作为具有多个子节点的节点：
  ```
     list
    /  |  \
   1   2   3
  ```
不再需要括号和分号。

* 在 `fun x -> 42` 中，`fun` 关键字和 `->` 标点符号分隔
来自周围代码的函数的参数和主体。一棵树可以
  将其表示为具有两个子节点的节点：
  ```
    function
    /     \
   x       42
  ```
不再需要关键字和标点符号。

因此，AST 代表了一个更容易理解的级别的程序结构。

**语义分析。** 在语义分析期间，编译器会根据它所实现语言的规则，
检查程序是否有意义。最常见的语义分析是类型检查：
编译器分析程序中所有表达式的类型，以查看是否存在类型错误。
通常类型检查
需要生成一个称为“符号表”的数据结构来映射
标识符（例如变量名）及其类型。当进入新的作用域时，
符号表通过新的绑定进行了扩展，这些新的绑定可能会影响旧的绑定；
当退出作用域时，新的绑定将被删除，从而恢复旧的绑定
绑定。因此，符号表混合了字典和栈这两种数据结构的特征。

除了类型检查之外，还有其他类型的语义分析。示例
包括以下内容：

* 检查 OCaml 模式匹配的分支是否详尽，
* 检查 C `break` 关键字是否出现在循环体中，以及
* 检查标记为 `final` 的 Java 字段是否最终在构造函数中被初始化。

你可以把解析看作“检查程序是否有意义”的一部分&mdash;这也是我们刚刚对语义分析的定义。
所以，解析和语义分析之间的区别更多是为了方便：解析完成足够的工作来生成 AST，
语义分析则完成剩余工作。

有时甚至需要语义分析才能完全确定 AST 的含义
应该是！例如，考虑类似 C 语言中的表达式 `(foo) - bar`
语言。可能是：

* 变量 `bar` 的一元否定，转换为类型 `foo`，或
* 使用操作数 `foo` 和 `bar` 进行二进制减法运算，其中
括号是多余的。

直到完成足够的语义分析以确定 `foo` 是否是
变量名或类型名，编译器不知道要生成哪个 AST。
在这种情况下，解析器通常会生成一个 AST，其中某些树
节点代表不明确的语法，然后语义分析阶段会重写这棵树，使其含义明确。

**翻译为中间表示。** 经过语义分析，
编译器*可以*立即翻译 AST（用符号表增强）
转换成目标语言。但是如果同一个编译器想要生成输出
多个目标（例如，对于 x86 和 ARM 和 MIPS），这需要定义一个
从 AST 到每个目标的翻译。在实践中，编译器
通常不这样做。相反，他们首先将 AST 转换为
中间表示 (IR)。可以把 IR 看作对许多汇编语言的一种抽象。
许多源语言（例如 C、Java、OCaml）都可以
翻译成相同的 IR，并从该 IR 输出许多目标语言（例如，
x86、ARM、MIPS）都可以生成。

IR 语言通常具有“抽象机器指令”来完成
概念上简单的任务：从内存加载或存储到内存，执行二进制
操作、调用和返回以及跳转到其他指令。
抽象机通常有无限数量的寄存器可用于
使用，就像源程序可以有无限数量的变量一样。真实
然而，机器的寄存器数量是有限的，这是一种方式
IR 是一种抽象。

**目标代码生成。** 编译的最后阶段是从 IR 生成目标代码。
此阶段通常涉及选择具体的机器指令（例如 x86 操作码），
并确定哪些变量将被存储在内存中（访问速度慢），
哪些变量将被存储在处理器寄存器中（访问速度快但数量有限）。
作为代码生成的一部分，编译器因此会尝试“优化”目标代码的性能。一些
优化的例子包括：

* 如果可以证明数组边界检查能够成功，则消除它们；
* 消除冗余计算；
* 适当地用函数本身的主体替换函数调用，并在参数上实例化，
  以消除调用和返回的开销；以及
* 重新排序机器指令，以便（例如）在真正需要结果之前就开始较慢的内存读取，
  并在等待读取结果的同时执行其他指令。

**阶段组。** 编译阶段可以分为两到三个部分：

* 编译器的“前端”执行词法分析、解析和语义分析。
它生成 AST 和关联的符号表，并将 AST 转换为 IR。

* 编译器的*中间端*（如果存在）在 IR 上运行。通常
这涉及执行独立于目标语言的优化。

* 编译器的*后端*进行代码生成，包括进一步的
优化。

**解释阶段。** 解释器的工作方式类似于编译器前端（也可能包括中端）。
也就是说，解释器进行词法分析、语法分析和
语义分析。然后它可能会立即开始执行 AST，或者它
可能会将 AST 转换为 IR 并开始执行 IR。

在本书的其余部分，我们将重点关注解释器。我们会忽略
IR 和代码生成，而是研究如何直接执行 AST。
