# Big-Oh 记号

“高效”是什么意思？康奈尔大学教授 Jon Kleinberg 和 Éva Tardos 在其教科书《算法设计》（2006）第 2 章中有精彩的阐释。本附录是从函数式编程视角对该内容的总结与诠释。
最终结论是：如果一个算法在输入规模 $n$ 上的最坏情况运行时间为 $O(n^d)$
（对某个常数 $d$），则认为该算法是“高效的”。但我们需要经过几个步骤才能建立这一定义。

## 算法和效率，尝试 1

定义效率的天真的尝试可能会如下进行：

**尝试 1：** 如果算法在实现后，对特定输入实例只花费少量时间，
则该算法是高效的。

但这个定义存在很多问题，例如：

- 低效的算法可以在小型测试用例上快速运行。

- 快速处理器和优化编译器可能导致低效算法运行
迅速地。

- 当编码马虎时，高效的算法可能会运行缓慢。

- 某些输入实例比其他实例更难。

- 小输入上的效率并不意味着大输入上的效率。

- 不同客户的耐心程度不同；对我来说快的，对你来说可能很慢。

**第 1 课：** 从这次尝试中学到的一个教训是：用时钟时间来衡量算法效率并不合理。
我们需要一个合理地独立于硬件、编译器和其他软件的指标。
也许一个好的衡量标准，是计算求值期间执行的“步骤”数量，而不是时钟时间。

但是，现在我们遇到了一个新问题：我们应该如何定义“步骤”？它需要是
与机器无关的东西。它应该以某种方式代表一个原始单位
计算。有很大的灵活性。以下是一些常见的选择：

- 在伪代码中，我们可能将步骤视为一行。

- 在命令式语言中，我们可能会计算赋值、数组索引、指针
解引用和算术运算作为步骤。

- 在 OCaml 中，我们可以计算函数或运算符应用程序、let 绑定，以及
选择 `if` 或 `match` 的分支作为步骤。

事实上，所有这些“步骤”可能确实需要不同的时间。
但实际上，这并不重要。

**第 2 课：** 我们从尝试 1 中学到的另一个教训是：在
特定的输入实例不是正确的指标。我们需要一个能够
预测任何输入实例的运行时间。所以不要使用特定的
输入（例如，数字、矩阵或文本文档），最好
使用输入的*规模*（例如，表示该值所需的位数
number，或者矩阵中的行数和列数，或者字节数
在文档中）作为度量。

但我们又遇到了一个新问题：如何定义“规模”？正如刚刚给出的例子所示，
规模应该衡量某个输入相对于其他输入有多大。
在数据结构的语境中，最常见的规模度量也许只是数据结构维护的元素数量：
列表中的节点数，或图中的节点数和边数等。

算法是否可能在两个“规模”相同的输入上运行不同时间？
当然。例如，将矩阵乘以全零可能会更快
比乘以任意数字。但在实践中，规模通常比
精确的输入更有用。

**第 3 课：** 我们从尝试 1 中学到的第三个教训是“少量
时间”这个词太相对了。我们想要一个相当客观的指标，
而不是依赖于什么构成“小”的主观概念。

一种不错的想法是，一种有效的算法需要击败
*暴力搜索*。这意味着一一列举所有答案，检查
各自看看是否正确。例如，暴力排序算法
会枚举列表的每个可能的排列，检查是否
它是输入列表的排序版本。这是一个糟糕的排序算法！
当然，快速排序胜过它。

暴力搜索是几乎所有算法的简单、愚蠢的解决方案
问题。但它需要枚举一个巨大的空间。事实上，一个
指数规模的空间。所以更好的想法是做少于指数级的
工作。什么小于指数（例如 $2^n$）？一种可能性是多项式
（例如，$n^2$）。

一个直接的反对意见可能是多项式有各种大小。例如，
$n^{100}$ 比 $n^2$ 大得多。还有一些非多项式，例如
$n^{1 +.02 (\log n)}$，可能足以击败指数。
但在实践中，多项式似乎确实工作得很好。

## 算法和效率，尝试 2

结合尝试 1 中的经验教训 1 到 3，我们进行了第二次尝试
定义效率：

**尝试 2：** 如果算法执行的最大步骤数是其输入规模的多项式，
则该算法是高效的。

注意这三个想法是如何结合在一起的：步骤数、规模、多项式。

但如果尝试使用这个定义，它仍然不完美。为最大执行步骤数给出精确公式，
可能会非常繁琐。例如，在另一本算法教科书中，作者推导出下面这个多项式，
用于表示插入排序伪代码实现的执行步骤数：

$$
c_1 n + c_2 (n - 1) + c_4 (n - 1) + c_5 \sum_{j=2}^{n} t_j + c_6 \sum_{j=2}^{n} (t_j - 1) + c_7 \sum_{j=2}^{n} (t_j - 1) + c_8 (n - 1)
$$

我们不需要解释所有变量的含义。它太复杂了。我们同情那位必须处理这个公式的研究生。

```{note}
插入排序运行时间的公式来自 Cormen、Leiserson、Rivest 和 Stein 的
*算法导论*（第 3 版，2009 年）。我们不是在取笑他们；
他们也会告诉你，这样的公式太复杂。
```

像这样精确的执行边界很难找到，而且有点
无意义的。如果 Java 伪代码需要 25 个步骤，但编译为
RISC-V需要250步，这个精度有用吗？

在某些情况下，是的。如果你正在构建驾驶飞机或控制飞机的代码
对于核反应堆，你可能实际上关心精确、实时的保证。

但除此之外，我们最好能识别出广泛的算法类别
具有类似的性能。而不是说算法运行在

$$1.62 n^2 + 3.5 n + 8$$

步骤，只是说它以 $n^2$ 步骤运行怎么样？也就是说，我们可以忽略
*低阶项*和最高阶项的*常数因子*。

我们忽略低阶项，因为我们想要大局观。算法效率为
所有这些都是关于解释当输入变得非常大时算法的性能。
我们不太关心小输入。低阶项在关注大规模输入时并不重要。
下表显示了步数与输入规模 N 的关系，假设每一步需要 1 微秒。
“很长”意味着更多
比宇宙中原子的估计数量还要多。

|           |  $N$ | $N$  $N^2$ |  $N^3$ |  $2^N$ |
 |:----------|:-----:|:--------:|:--------:|:--------:| 
 |N=10 |< 1 秒|< 1 秒 |< 1 秒 | < 1 秒 |
 |N=100 |< 1 秒|< 1 秒 |  1秒|1017年|
 |N=1,000 |< 1 秒|  1 秒 | 18 分钟 |很长|
 |N=10,000 |< 1 秒|  2 分钟 | 12 天 |很长|
 |N=100,000 |< 1 秒|  3小时| 32年|很长|
 |N=1,000,000|  1秒| 12天|104年|很长|

正如你所看到的，当输入变大时，表中各列之间会出现巨大差异。
我们不妨忽略低阶项，因为在考虑大规模输入时，它们完全被最高阶项主导。

常数因子呢？我当前的笔记本电脑可能会快 2 倍（也就是说，
常数因子 2) 比我几年前买的那个要大，但这不是一个
该算法的有趣属性。同样，伪代码中的 $1.62 n^2$ 步骤
可能是 $1620 n^2$ 条汇编步骤（即常数因子 1000），但是
这又不是算法的一个有趣的属性。那么，我们真的应该
关心一种算法是否比另一种算法花费 2 倍或 1000 倍的时间，如果它只是一个
常数因子？

答案是：也许吧。实际代码中的性能调优是为了获得
让常数变小。如果你让某段代码快了一倍，你的雇主可能会很高兴！
但这与**算法**无关。当我们衡量算法效率时，常数因子实际上并不重要。

因此，所有这些都说明，我们应该用**不精确的抽象**来衡量运行时间。
我们可以直接写 $n^2$，而不是 $1.62 n^2 + 3.5 n + 8$。
不精确抽象对你来说并不新鲜。你可能会写 $\pm 1$ 来不精确地
抽象出 1 以内的量。在计算机科学中，你已经知道我们使用
Big-Oh 表示法作为不精确的抽象：$1.62 n^2 + 3.5 n + 8$ 是 $O(n^2)$。

## Big-Ell 表示法

在回顾 Big-Oh 表示法之前，让我们先从一些简单的东西开始
以前可能没见过：Big-Ell 记号。

Big-Ell 是对“小于或等于”（<u>l</u>ess than or equal to）某个数的自然数集合的不精确抽象。
它的定义如下：

$$
L(n) = \{ m \mid 0 \leq m \leq n\}
$$

其中 $m$ 和 $n$ 是自然数。即 $L(n)$ 代表所有
小于或等于 $n$ 的自然数。例如，$L(5) = \{0, 1, 2, 3, 4,
5\}$。

你能用 Big-Ell 做算术吗？例如，$1 + L(5)$ 是什么？
老实说，这不是一个适定的问题：我们认为加法是一种运算
是在整数上定义的，而不是在整数集上定义的。但合理的
$1 + \{0, 1, 2, 3, 4, 5\}$ 的解释可能是做加法
集合中的每个元素，产生 $\{1, 2, 3, 4, 5, 6\}$。请注意 $\{1, 2, 3,
4, 5, 6\}$ 是 $\{0, 1, 2, 3, 4, 5, 6\}$ 的真子集，后者是
$L(6)$。所以我们可以说$1 + L(5) \subseteq L(6)$。我们甚至可以说
$1 + L(5) \subseteq L(7)$，但它不是*紧*：前一个子集关系
包含尽可能少的额外元素，而后者是“松散”的
不必要地包括额外的。

有关 Big Ell 的更多信息，请参阅*具体数学*，第 9 章，1989 年，作者：Graham，
高德纳和帕塔什尼克。

## Big-Oh 记号

如果你了解 Big-Ell，并且了解函数式编程，那么这里是
一些好消息：你可以轻松理解 Big-Oh。

让我们通过几个步骤来构建 Big-Oh 的定义。我们将从版本开始
1，我们将其写为 $O_1$。它基于 $L$：

- $L(n)$ 表示任何小于或等于 a 的**自然数**
**自然数** $n$。

- $O_1(g)$ 表示任何小于或等于的**自然函数**
到 **自然函数** $g$。

*自然函数*只是自然数上的函数；也就是说，它的类型是
$\mathbb{N} \rightarrow \mathbb{N}$。

我们对 $O_1$ 所做的就是从 **自然数** 升级到 **自然数
函数**。所以 Big-Oh 版本 1 只是 Big-Ell 的*高阶*版本。
怎么样！

当然，我们需要弄清楚函数小于意味着什么
另一个函数。这是一个合理的形式化：

**Big-Oh 版本 1：** $O_1(g) = \{ f \mid \forall n \mathrel{.} f(n) \leq g(n)\}$

例如，考虑将其输入加倍的函数。在数学课本上，
该函数可以写为 $g(n) = 2n$。在 OCaml 中我们会写
`let g n = 2 * n` 或 `let g = fun n -> 2 * n` 或只是匿名
`fun n -> 2 * n`。在数学中，相同的匿名函数可以写成
lambda 表示法为 $\lambda n . 2n$。继续使用 lambda 表示法，我们有：

$$
O_1(\lambda n . 2n) = \{ f \mid \forall n \mathrel{.} f(n) \leq 2n \}
$$

因此

- $(\lambda n . n) \in O_1(\lambda n . 2n)$，

- $(\lambda n . \frac{n}{2}) \in O_1(\lambda n . 2n)$，但是

- $(\lambda n . 3n) \notin O_1(\lambda n . 2n)$。

接下来，回想一下，在定义算法效率时，我们想忽略
常数因子。$O_1$ 对此没有帮助。我们真正想要的是让所有
这些函数：

- $(\lambda n . n)$

- $(\lambda n . 2n)$

- $(\lambda n . 3n)$

都位于 $O(\lambda n . n)$ 中。

为此，我们定义 $O_2$ 来忽略常数因子：

**Big-Oh 版本 2：** $O_2(g) = \{ f \mid \exists c \gt 0 \forall n
\mathrel{.} f(n) \leq c g(n)\}$

存在量化的正常数 $c$ 让我们可以把函数 $g$
“抬高”为所需的任何常数倍。例如，

$$
O_2(\lambda n . n^3) = \{ f \mid \exists c \gt 0 \forall n \mathrel{.} f(n) \leq c n^3 \}
$$

因此 $(\lambda n . 3n^3) \in O_2(\lambda n . n^3)$，因为 $3 n^3 \leq
c n^3$，只要取 $c = 3$、$c = 4$ 或任何更大的 $c$。

最后，请记住，我们不关心小输入：分析算法效率时，我们关注大规模情况。
对于小输入，算法恰好快一点或慢一点并不重要。
事实上，我们可以为这些小输入硬编码一个查找表，如果
算法对他们来说太慢了！真正重要的是性能
大规模输入。

为此，我们定义 $O_3$ 来忽略小输入：

**Big-Oh 版本 3：** $O_3(g) = \{ f \mid \exists c \gt 0 \exists n_0 \gt
0 \forall n \geq n_0 \mathrel{.} f(n) \leq c g(n)\}$

存在量化的正常数 $n_0$ 让我们可以“忽略”所有
规模不超过它的输入。例如，

$$
O_3(\lambda n . n^2) = \{ f \mid \exists c \gt 0 \exists n_0 \gt 0 \forall n \geq n_0 \mathrel{.} f(n) \leq c n^2\}
$$

因此 $(\lambda n . 2n) \in O_3(\lambda n . n^2)$，因为 $2n \leq c
n^2$，只要取 $c = 2$ 且 $n_0 = 2$。请注意，通过选择 $n_0 = 2$，
我们忽略了 $\lambda n . 2n$ 在 $n = 1$ 处暂时有点太大的事实。
这就是忽略“小”输入的力量。

## Big-Oh，到此为止

第 3 版是 Big-Oh 的正确定义。  我们在这里重复一遍，真的：

**Big-Oh：** $O(g) = \{ f \mid \exists c \gt 0 \exists n_0 \gt 0 \forall n \geq n_0 \mathrel{.} f(n) \leq c g(n)\}$

这是你应该了解的最终重要版本。但不要只是记住它。
如果你理解我们在这里给出的推导，你将能够重新创建它
任何需要的时候都可以从头开始。

Big-Oh 被称为*渐近上界*。如果 $f \in O(g)$，
则 $f$ 的效率至少不比 $g$ 渐近更差，但也可能比 $g$ 更高效。

## Big-Oh 记号的注意事项

**警告 1.** 因为它是一个上限，所以我们总是可以夸大 Big-Oh
语句：例如，如果 $f \in O(n^2)$，则也是 $f \in O(n^3)$，并且 $f \in
O(2^n)$ 等。但我们的目标始终是给出*紧*上限，无论我们是否
明确地说或不说。所以当被问到算法的运行时间是多少时
也就是说，你应该始终给出尽可能紧的 Big-Oh 界。

**警告 2.** 大多数作者不是使用 $O(g) = \{ f \mid \ldots \}$
而是写 $O(g(n)) = \{ f(n) \mid \ldots \}$。他们并不是真正的意思
$g$ 应用于 $n$。它们意味着在输入 $n$ 上参数化的函数 $g$ 但不是
尚未应用。这是严重误导的，通常是由于不
理解匿名函数。这个故事的寓意是：更多的人需要
研究函数式编程。

**警告 3.** 而不是 $\lambda n . 2n \in O(\lambda n . n^2)$ 几乎所有
作者写作 $2n = O(n^2)$。这是对符号的严重滥用，
本不应该被允许，却永久地进入了计算机科学的集体意识。
标准辩护是这里的 $=$ 应读作“是”而不是“等于”。
这显然站不住脚，甚至提出这种辩护的人通常也会礼貌地承认它很牵强。有时
我们陷入了祖先的错误之中。这是其中之一。
小心这种“单向平等”，如果有机会的话，
教你的（智力）孩子做得更好。

## 算法和效率，尝试 3

我们来回顾一下。  我们定义效率的第一次尝试是：

**尝试 1：** 如果算法在实现后，对特定输入实例只花费少量时间，
则该算法是高效的。

通过用步骤数替代时间、用输入规模替代特定输入实例、
用多项式替代“少量”，我们将其改进为：

**尝试 2：** 如果算法执行的最大步骤数是其输入规模的多项式，
则该算法是高效的。

这确实是一个非常好的定义。  但使用 Big-Oh 表示法
让它更具体一点，我们可以进行第三次也是最后一次尝试：

**尝试 3：** 如果算法在规模为 $n$ 的输入上的最坏情况运行时间是
$O(n^d)$（对某个常数 $d$），则该算法是高效的。

通过“最坏情况运行时间”，我们的意思与“最大数量”相同
执行步骤”，只是用不同且可能更常见的词语表达。
最坏的情况是执行时间最长。 “时间”是一个常见的委婉说法
这里指执行步骤，用来强调我们要考虑计算需要多长时间。

*空间*是最常见的另一个需要考虑的效率特征。算法
在需要常数或线性空间时，效率可能更高或更低。
你已经通过尾递归和列表熟悉了这一点
OCaml。
