5.3. 模块和顶层#

备注

下面的视频使用旧版构建系统 ocamlbuild，而不是新的 构建系统，Dune。一些细节随着Dune的变化而改变，如 下面的文字。

有几个涉及模块和顶层的语用学： 掌握如何有效地结合使用两者很重要。

5.3.1. 加载编译模块#

编译 OCaml 文件会生成一个与该文件同名的模块，但是 第一个字母大写。这些编译好的模块可以加载到 顶层使用 #load。

例如，假设你创建一个名为 mods.ml 的文件，并输入以下内容 里面的代码：

let b = "bigred"
let inc x = x + 1
module M = struct
  let y = 42
end

请注意，周围没有 module Mods = struct ... end 。代码位于 可以说是文件的最顶层。

然后假设你输入 ocamlc mods.ml 来编译它。新创建的其中之一 文件是mods.cmo：这是一个c编译的module object文件， 又名字节码。

你可以使用以下命令使该字节码可在顶层使用 指令。回想一下，指令前面需要 # 字符。 它不是提示的一部分。

# #load "mods.cmo";;

该指令加载在 mods.cmo 中找到的字节码，从而创建一个模块 名为 Mods 可供使用。就好像你已经输入了这个一样 代码：

module Mods = struct
  let b = "bigred"
  let inc x = x + 1
  module M = struct
    let y = 42
  end
end

module Mods :
  sig val b : string val inc : int -> int module M : sig val y : int end end

因此，这两个表达式都将成功求值：

Mods.b;;
Mods.M.y;;

- : string = "bigred"

- : int = 42

但这会失败：

inc

File "[3]", line 1, characters 0-3:
1 | inc
    ^^^
Error: Unbound value inc
Hint: Did you mean incr?

它失败是因为 inc 位于 Mods 的命名空间中。

Mods.inc

- : int -> int = <fun>

当然，如果打开模块，可以直接命名为inc：

open Mods;;
inc;;

- : int -> int = <fun>

5.3.2. Dune#

Dune 提供了一个命令，可以更轻松地启动带有库的 utop 已加载。假设我们将此Dune文件添加到与 mods.ml 相同的目录中：

(library
 (name mods))

这告诉Dune从 mods.ml （以及任何其他 同一目录中的文件（如果存在）。 然后我们可以运行这个命令 启动 utop 并加载该库：

$ dune utop

现在我们可以立即访问 Mods 的组件，而无需发出 #load 指令：

Mods.inc

- : int -> int = <fun>

如果你愿意，dune utop 命令接受目录名称作为参数 在源代码的特定子目录中加载库。

5.3.3. 初始化顶层#

如果你正在对某个特定模块进行大量测试，那么这可能会很烦人 每次启动 utop 时都必须输入指令。你确实想要初始化 顶层在启动时包含一些代码，这样你就不必保留 输入该代码。

解决方案是在工作目录中创建一个文件并调用该文件 .ocamlinit。请注意，该文件名前面的 . 是必需的，并且 使其成为 hidden file ，不会出现在目录列表中，除非 明确请求（例如，使用 ls -a）。 .ocamlinit 中的所有内容都将是 加载时由 utop 处理。

例如，假设你在同一目录中创建一个名为 .ocamlinit 的文件 作为 mods.ml，并在该文件中放入以下代码：

open Mods;;

现在使用 dune utop 重新启动 utop。 Mods 中定义的所有名称都已经 处于范围内。例如，这些都会成功：

inc;;
M.y;;

- : int -> int = <fun>

- : int = 42

5.3.4. 引入库#

假设你想在 utop 中试验一些 OUnit 代码。你不能 实际上打开它：

open OUnit2;;

File "[8]", line 1, characters 5-11:
1 | open OUnit2;;
         ^^^^^^
Error: Unbound module OUnit2
Hint: Did you mean Unit?

问题是 OUnit 库尚未加载到 utop 中。它可以 符合以下指令：

#require "ounit2";;

现在你可以成功加载自己的模块而不会出现错误。

open OUnit2;;

5.3.5. load 与 use#

#load-编译的模块文件和 #use-ing 未编译的源文件。前者加载字节码并使其 可供使用。例如，加载 mods.cmo 导致 Mod 模块被 可用，我们可以使用 Mod.b 等表达式访问其成员。 后者（#use）是文本包含：就像输入 文件直接进入顶层。因此，使用 mods.ml 不会**导致 Mod 模块可用，并且可以访问文件中的定义 直接，例如 b。

例如，在下面的交互中，我们可以直接引用b，但是 不能使用限定名称 Mods.b：

# #use "mods.ml"

# b;;
val b : string = "bigred"

# Mods.b;;
Error: Unbound module Mods

而在这种互动中，情况正好相反：

# #directory "_build";;
# #load "mods.cmo";;

# Mods.b;;
- : string = "bigred"

# b;;
Error: Unbound value b

因此，当你使用顶层来试验代码时，通常会 更好地使用 #load 而不是 #use。 #load 指令准确 反映了你的模块如何相互交互以及与外部世界交互。
