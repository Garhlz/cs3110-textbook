# 类型推断

OCaml 和 Java 是“静态类型”语言，这意味着每个绑定都有一个
在*编译时*&mdash;也就是程序执行之前&mdash;确定的类型。
类型检查器是一个编译时过程，用于接受或拒绝程序。相比之下，JavaScript 和 Ruby 则
动态类型语言；绑定的类型未提前确定
确定。诸如将 42 绑定到 `x` 然后将 `x` 视为字符串的计算
因此要么导致运行时错误，要么导致运行时转换
类型。

与 Java 不同，OCaml 是“隐式类型”，这意味着程序员很少需要
写下绑定的类型。这通常很方便，尤其是在
高阶函数。 （尽管有些人不同意这是否使
代码更容易还是更难阅读。）但隐式类型绝不会改变这个事实：
OCaml 是静态类型的。相反，类型检查器必须更多
复杂，因为它必须推断出如果程序员把所有类型注释都写出来，
那些注释会是什么。原则上，类型推断和
类型检查可以是单独的过程（推断器可以找出
类型，然后检查器可以确定程序是否类型正确），但是
在实践中，它们通常被合并到一个名为 *type 的单个过程中
重建*。

## OCaml 类型重建

{{ video_embed | replace("%%VID%%", "_yDo9Q9EOHY")}}

在非常高的层面上，OCaml 的类型重建算法的工作原理如下：

- 按顺序确定定义的类型，使用前面定义的类型来推断后面定义的类型。
  （这就是为什么你不能在 OCaml 程序中先使用一个名称，再绑定它。）

- 对于每个 `let` 定义，分析定义以确定*约束*
关于它的类型。例如，如果推理器看到 `x + 1`，则得出结论：
  `x` 必须具有类型 `int`。它会从函数应用、模式匹配等地方收集类似约束。
  可以把这些约束看作一个系统，
  就像代数中可能有的方程一样。

- 使用该方程组来求解所定义的名称的类型。

OCaml 类型重建算法尝试永远不会拒绝以下程序：
如果程序员写下了类型，就可以进行类型检查。它还尝试
永远不要接受不可能进行类型检查的程序。还有一些比较隐晦的
语言的某些部分有时可能需要类型注释，或者在
最不方便时（请参阅*现实世界 OCaml* 第 22 章“类型推断”，了解
示例）。但对于你编写的大多数代码来说，类型注释确实完全是
可选的。

因为继续写“类型重建算法”会很冗长
OCaml 和其他函数式语言使用”，我们将该算法称为 HM。那
整个编程语言文献中都使用这个名称，因为
算法由 Roger <u>H</u>indley 和 Robin 独立发明
<u>M</u>ilner.

HM 已经被很多人多次重新发现。Curry 非正式地使用了它
20 世纪 50 年代（甚至可能是 1930 年代）。他于1967年正式写成（发表
1969）。 Hindley于1969年独立发现它；莫里斯，1968 年；和米尔纳
1978 年。在逻辑领域，类似的想法可能可以追溯到 Tarski
在20世纪20年代。在评论这段历史时，欣德利写道：

> 这个不断重新发现的故事一定有其寓意；也许有人
> 沿着这条线应该已经学会了阅读。或者别人学写作。

虽然我们还没有看到 HM 算法，但你可能不会感到惊讶
要知道它通常非常有效&mdash;你可能从来没有必要这样做
等待顶层打印推断出的程序类型。在实践中，
它以近似线性的时间运行。但理论上来说，有一些很奇怪的地方
可能导致其运行时间爆炸的程序。 （从技术上来说，它是
指数时间。）为了好玩，尝试在 utop 中键入以下代码：

```ocaml
# let b = true;;
# let f0 = fun x -> x + 1;;
# let f = fun x -> if b then f0 else fun y -> x y;;
# let f = fun x -> if b then f else fun y -> x y;;
# let f = fun x -> if b then f else fun y -> x y;;
(* keep repeating that last line *)
```

你会看到类型变得越来越长，最终（大约 20
重复等）类型推断将导致显着的延迟。

## 基于约束的类型推断

让我们从这个小东西开始构建 HM 类型推断算法
语言：

```text
e ::= x | i | b | e1 bop e2
    | if e1 then e2 else e3
    | fun x -> e
    | e1 e2

bop ::= + | * | <=

t ::= int | bool | t1 -> t2
```

该语言是 SimPL，加上 lambda 演算，减去 `let` 表达式。它
结果 `let` 表达式增加了一层额外的复杂性，所以我们来
稍后再回复他们。

由于这种语言中的匿名函数没有类型注释，因此我们有
推断参数 `x` 的类型。例如，

- 在 `fun x -> x + 1` 中，参数 `x` 必须具有类型 `int` 因此该函数具有
输入 `int -> int`。

- 在 `fun x -> if x then 1 else 0` 中，参数 `x` 必须具有类型 `bool` 因此
函数的类型为 `bool -> int`。

- 函数 `fun x -> if x then x else 0` 是不可输入的，因为它会
要求 `x` 具有类型 `int` 和 `bool`，这是不允许的。

**语法简化。** 我们可以将 `e1 bop e2` 视为语法糖
`( bop ) e1 e2`。也就是说，我们将中缀二元运算符视为前缀函数
应用程序。让我们为 *names* 引入一个新的语法类 `n` ，它
概括标识符和运算符。这会将语法更改为：

```text
e ::= n | i | b
    | if e1 then e2 else e3
    | fun x -> e
    | e1 e2

n ::= x | bop

bop ::= ( + ) | ( * ) | ( <= )

t ::= int | bool | t1 -> t2
```

我们已经知道这些内置运算符的类型：

```text
( + ) : int -> int -> int
( * ) : int -> int -> int
( <= ) : int -> int -> bool
```

这些类型已给出；我们不必推断它们。它们是最初的一部分
静态环境。在 OCaml 中，这些运算符名称稍后可能会被隐藏
具有不同类型的值，但在这里我们不必担心，因为
我们还没有 `let`。

*你*如何在心里推断出 `fun x -> 1 + x` 的类型，或者更确切地说，
`fun x -> ( + ) 1 x`？现在它是自动的，但我们可以将其分解为
件：

- 从 `x` 开始，具有一些未知类型 `t`。
- 请注意，已知 `( + )` 的类型为 `int -> (int -> int)`。
- 因此它的第一个参数必须具有类型 `int`。  `1` 就是这样做的。
- 它的第二个参数也必须具有 `int` 类型。所以`t = int`。那是一个
`t` 上的_约束_。
- 最后，函数体还必须具有类型 `int`，因为这是
返回类型为 `( + )`。
- 因此，整个函数的类型必须是`t -> int`。
- 由于 `t = int`，该类型为 `int -> int`。

类型推断算法遵循生成未知类型的相同思想，
收集它们的约束，并使用约束来求解类型
的表达式。

{{ video_embed | replace("%%VID%%", "hrl9Q68dIfQ")}}

我们引入一个新的四元关系`env |- e : t -| C`，它应该是
如下所示：“在环境 `env` 中，表达式 `e` 被推断为具有类型
`t` 并生成约束集 `C`。”约束是以下形式的方程
`t1 = t2` 对于任何类型 `t1` 和 `t2`。

如果我们将关系视为类型推断函数，则冒号中的
中间将输入与输出分开。输入是 `env` 和 `e`：我们
想知道环境 `env` 中 `e` 的类型是什么。该函数返回
作为输出类型 `t` 和约束 `C`。

关系中间的 `e : t` 大约是你在
toplevel：你输入一个表达式，它会告诉你类型。但围绕那是
你不可见的环境和约束集 `env |- ... -| C`。
因此，外部的十字转门显示了类型推断的部分，即
顶层没有。

{{ video_embed | replace("%%VID%%", "NkAt9eApGSw")}}

推理最简单的部分是常数：

```text
env |- i : int -| {}

env |- b : bool -| {}
```

已知任何整型常量 `i`，例如 `42`，都具有类型 `int`，并且
没有产生约束。  对于布尔常量也是如此。

推断名称的类型需要在环境中查找：

```text
env |- n : env(n) -| {}
```

不产生约束。

如果名称未绑定在环境中，则无法键入表达式。
这是一个未绑定名称错误。

其余规则的核心与我们看到的类型检查规则相同
以前，但它们每个都会生成一个_类型变量_，并且可能还生成一些
该类型变量的约束。

**如果。**

{{ video_embed | replace("%%VID%%", "0EHUTbWnYWw")}}

以下是 `if` 表达式的规则：

```text
env |- if e1 then e2 else e3 : 't -| C1, C2, C3, t1 = bool, 't = t2, 't = t3
  if fresh 't
  and env |- e1 : t1 -| C1
  and env |- e2 : t2 -| C2
  and env |- e3 : t3 -| C3
```

为了推断 `if` 的类型，我们推断每个的类型 `t1`、`t2` 和 `t3`
它的子表达式，以及它们的任何约束。我们无法控制
这些类型可能是什么；这取决于程序员写的内容。但我们确实
知道守卫的类型必须是`bool`。所以我们生成一个约束
`t1 = bool`。

此外，我们知道两个分支必须具有相同的类型 &mdash; 但是，我们
事先不知道那种类型可能是什么。所以，我们发明了一个 _fresh_ 类型
变量 `'t` 代表该类型。如果类型变量从未出现过，则它是新鲜的
在类型推断期间在其他地方使用。因此，选择一个新的类型变量
只是意味着选择一个不可能与任何其他名称混淆的新名称
程序中的名字。我们返回 `'t` 作为 `if` 的类型，并记录两个
约束 `'t = t2` 和 `'t = t3` 表示两个分支都必须具有该约束
类型。

因此，我们需要将类型变量添加到类型语法中：

```text
t ::= 'x | int | bool | t1 -> t2
```

一些示例类型变量包括 `'a`、`'foobar` 和 `'t`。最后，`t`
是一个标识符，而不是元变量。

这是一个例子：

```text
{} |- if true then 1 else 0 : 't -| bool = bool, 't = int
  {} |- true : bool -| {}
  {} |- 1 : int -| {}
  {} |- 0 : int -| {}
```

生成的完整约束集是
`{}, {}, {}, bool = bool, 't = int, 't = int`，但这当然简化为
只是`bool = bool, 't = int`。从该约束集我们可以看到类型
`if true then 1 else 0` 必须是 `int`。

**匿名函数。**

{{ video_embed | replace("%%VID%%", "y2Y2aRnxncE")}}

由于 `x` 上没有类型注释，因此必须推断其类型：

```text
env |- fun x -> e : 't1 -> t2 -| C
  if fresh 't1
  and env, x : 't1 |- e : t2 -| C
```

我们引入一个新的类型变量 `'t1` 来代表 `x` 的类型，并推断
`x : 't1` 环境下主体 `e` 的类型。无论 `x` 在哪里
在 `e` 中使用，可能会导致生成涉及 `'t1` 的约束。那些
约束将成为 `C` 的一部分。

这是一个我们可以立即看到 `x : bool` 的函数，但是让我们开始工作
通过推论：

```text
{} |- fun x -> if x then 1 else 0 : 't1 -> 't -| 't1 = bool, 't = int
  {}, x : 't1 |- if x then 1 else 0 : 't -| 't1 = bool, 't = int
    {}, x : 't1 |- x : 't1 -| {}
    {}, x : 't1 |- 1 : int -| {}
    {}, x : 't1 |- 0 : int -| {}
```

函数的推断类型为 `'t1 -> 't`，约束为 `'t1 = bool`
和 `'t = int`。简化一下，该函数的类型是 `bool -> int`。

**函数应用。**

{{ video_embed | replace("%%VID%%", "2HRcvmQBWIM")}}

必须推断整个应用程序的类型，因为我们还不知道
有关任一子表达式类型的任何信息：

```text
env |- e1 e2 : 't -| C1, C2, t1 = t2 -> 't
  if fresh 't
  and env |- e1 : t1 -| C1
  and env |- e2 : t2 -| C2
```

我们为应用程序的类型引入了一个新的类型变量 `'t`
表达。我们使用推理来确定子表达式的类型和
他们碰巧产生的任何限制。我们添加一个新约束，
`t1 = t2 -> 't`，表示左侧`e1`的类型必须
是一个函数，它接受 `t2` 类型的参数并返回类型的值
`'t`。

令 `I` 为绑定布尔运算符的_初始环境_。让我们
推断 `( + )` 的部分应用的类型：

```text
I |- ( + ) 1 : 't -| int -> int -> int = int -> 't
  I |- ( + ) : int -> int -> int -| {}
  I |- 1 : int -| {}
```

从由此产生的约束中，我们看到

```text
int -> int -> int
=
int -> 't
```

从每个函数类型的左侧剥离 `int ->` ，
我们只剩下

```text
int -> int
=
't
```

因此，`( + ) 1` 的类型是 `int -> int`。

## 解决约束

{{ video_embed | replace("%%VID%%", "o1wT8FC9hpE")}}

解决一组约束意味着什么？由于约束是方程
就类型而言，这很像求解代数方程组。我们想要
求解这些方程中出现的变量的值。由
将这些值替换为变量，我们应该得到如下方程
两侧相同。例如，在代数中我们可能有：

```text
5x + 2y =  9
 x -  y = -1
```

求解该系统，我们会得到 `x = 1` 和 `y = 2`。  如果我们替换
`1` 对于 `x` 和 `2` 对于 `y`，我们得到：

```text
5(1) + 2(2) =  9
  1  -   2  = -1
```

这减少到

```text
 9 =  9
-1 = -1
```

在编程语言术语中（尽管可能不是高中代数），
我们说替换 `{1 / x}` 和 `{2 / y}` 一起*统一*该集合
方程组，因为它们使每个方程“联合”，使得它的左侧是
与其右侧相同。

{{ video_embed | replace("%%VID%%", "cNqPY5MSutM")}}

求解类型方程组是类似的。正如我们找到的数字一样
替换上面的变量，我们现在想要找到类型来替换类型
变量，从而统一方程组。

很像我们之前在替换模型求值时定义的替换，
我们将把“把类型变量 `'x` 映射到类型 `t`”的*类型替换*写作 `{t / 'x}`。
例如，`t1 {t2/'x}` 表示类型 `t1` 和 `t2`
替换 `'x`。

我们可以如下定义类型的替换：

```text
int {t / 'x} = int
bool {t / 'x} = bool
'x {t / 'x} = t
'y {t / 'x} = 'y
(t1 -> t2) {t / 'x} =  (t1 {t / 'x} ) -> (t2 {t / 'x} )
```

给定两个替换 `S1` 和 `S2`，我们写 `S1; S2` 来表示
替换是它们的*顺序组合*，定义为
如下：

```text
t (S1; S2) = (t S1) S2
```

顺序很重要。例如，`'x ({('y -> 'y) / 'x}; {bool / 'y}) ` 是
`bool -> bool`，而不是 `'y -> 'y`。我们可以建立越来越大的替代品
这边。

替换 `S` 可以应用于约束 `t = t'`。结果
`(t = t') S` 定义为 `t S = t' S`。所以我们只需将替换应用于
约束的两侧。

最后，可以将替换应用于一组 `C` 约束；结果
`C S` 是将 `S` 应用于每个单独约束的结果
`C`。

如果 `t_1 S` 产生相同的结果，则替换*统一*约束 `t_1 = t_2`
输入 `t_2 S`。例如，替换 `S = {int -> int / 'y}; {int / 'x}`
统一约束 `'x -> ('x -> int) = int -> 'y`，因为

```text
('x -> ('x -> int)) S
=
int -> (int -> int)
```

和

```text
(int -> 'y) S
=
int -> (int -> int)
```

如果 `S` 统一了每个约束，则替换 `S` 统一一组 `C` 约束
`C` 中的约束。

最后，我们可以准确地说解决一组约束意味着什么：我们
必须找到统一该集合的替换。也就是说，我们需要找到一个
从类型变量到类型的映射序列，使得该序列导致
约束中的每个方程设置为“统一”，这意味着它的左侧
和右侧变得相同。

为了找到统一约束集 `C` 的替换，我们使用一种算法
`unify`，定义如下：

- 如果 `C` 是空集，则 `unify(C)` 是空替换。

- 如果 `C` 包含至少一个约束 `t1 = t2` 以及可能的其他一些约束
约束 `C'`，则 `unify(C)` 定义如下：

- 如果 `t1` 和 `t2` 都是相同的简单类型 &mdash; 即，两者相同
      类型变量 `'x`，或两者 `int` 或两者 `bool`&mdash; 然后返回
      `unify(C')`。 *在这种情况下，约束不包含任何有用的内容
      信息，所以我们将其丢弃并继续。*

- 如果 `t1` 是类型变量 `'x` 并且 `'x` 没有出现在 `t2` 中，那么让
      `S = {t2 / 'x}`，并返回`S; unify(C' S)`。 *在这种情况下，我们是
      从方程组中消除变量 `'x` ，就像
      求解代数方程中的高斯消元法。*

- 如果 `t2` 是类型变量 `'x` 并且 `'x` 没有出现在 `t1` 中，那么让
      `S = {t1 / 'x}`，并返回`S; unify(C' S)`。 *这是一个消除
      就像之前的案例一样。*

- 如果 `t1 = i1 -> o1` 和 `t2 = i2 -> o2`，其中 `i1`、`i2`、`o1` 和 `o2`
      是类型，然后是 `unify(i1 = i2, o1 = o2, C')`。 *在这种情况下，我们打破了一个
      将约束分解为两个较小的约束并将这些约束相加
      返回以进一步统一。*

- 否则，失败。没有可能的统一者。

<!--
- 如果 `t = t0 * t1` 和 `t' = t'0 * t'1`，
      然后让 `C''` 成为 `C'` 与约束的并集
      `t0 = t'0` 和 `t1 = t'1`，并返回 `unify(C'')`。

- 如果 `t = (t0, ..., tn) tc` 和 `t' = (t'0, ..., t'n) tc` 对于某些
      类型构造函数 `tc`,
      然后让 `C''` 成为 `C'` 与约束的并集
      `ti = t'i`，并返回`unify(C'')`。
-->

在第二个和第三个子情况中，检查 `'x` 不应出现在
type 确保算法实际上消除了变量。否则，
该算法最终可能会重新引入变量而不是消除
它。

可以证明统一算法总是终止，并且
当且仅当统一符实际存在时它才会产生结果&mdash;也就是说，
当且仅当约束集有解。此外，该解决方案
算法产生的是*最通用的统一器*，从某种意义上说，如果
`S = unify(C)`和`S'`又统一了`C`，那么一定存在一些`S''`这样的
那个`S' = S; S''`。这样的 `S'` 不如 `S` 通用，因为它包含
`S''` 的额外替换。

## 整理类型推断

让我们回顾一下到目前为止我们所做的事情。我们从这种语言开始：

```text
e ::= n | i | b
    | if e1 then e2 else e3
    | fun x -> e
    | e1 e2

n ::= x | bop

bop ::= ( + ) | ( * ) | ( <= )

t ::= int | bool | t1 -> t2
```

然后我们引入了一种用于推断表达式类型的算法。那种类型
随之而来的是一系列限制。该算法表达为
关系 `env |- e : t -| C`。

接下来，我们介绍了求解约束集的统一算法。那
算法生成替换序列 `S` 作为输出，否则会失败。如果它
失败，则 `e` 不可输入。

为了完成类型推断并重建 `e` 的类型，我们只需计算 `t S` 即可。
也就是说，我们将解决方案应用于由 生成的类型 `t` 的约束
约束的产生。

让 `p` 成为这种类型。即 `p = t S`。可以证明 `p` 是
表达式的 *principal* 类型，这意味着如果 `e` 也具有 `t` 类型
任何其他 `t`，则存在替换 `S` 使得 `t = p S`。

例如，恒等函数 `fun x -> x` 的主体类型将是
`'a -> 'a`。但你也可以为该函数提供不太有用的类型
`int -> int`。我们所说的是 HM 将产生 `'a -> 'a`，而不是
`int -> int`。所以从某种意义上来说，HM实际上推断出了最“宽松”的类型，那就是
可以用于表达式。

**一个有效的示例。** 让我们推断以下表达式的类型：

```ocaml

```

```text
       I, f : 'a, x : 'b |- ( + ) x 1 : 'd -| 'c = int -> 'd, int -> int -> int = 'b -> 'c
         I, f : 'a, x : 'b |- ( + ) x : 'c -| int -> int -> int = 'b -> 'c
           I, f : 'a, x : 'b |- ( + ) : int -> int -> int -| {}
           I, f : 'a, x : 'b |- x : 'b -| {}
         I, f : 'a, x : 'b |- 1 : int -| {}
```

作为约束生成的结果，我们知道表达式的类型是
`'a -> 'b -> 'e`，其中

```text
'a = 'd -> 'e
'c = int -> 'd
int -> int -> int = 'b -> 'c
```

为了求解该方程组，我们使用统一算法：

```text
unify('a = 'd -> 'e, 'c = int -> 'd, int -> int -> int = 'b -> 'c)
```

第一个约束产生一个替换 `{('d -> 'e) / 'a}`，我们记录下来
作为解决方案的一部分，并将其应用于其余约束：

```text
...
=
{('d -> 'e) / 'a}; unify(('c = int -> 'd, int -> int -> int = 'b -> 'c) {('d -> 'e) / 'a})
=
{('d -> 'e) / 'a}; unify('c = int -> 'd, int -> int -> int = 'b -> 'c)
```

第二个约束的行为与第一个约束类似：

```text
...
=
{('d -> 'e) / 'a}; {(int -> 'd) / 'c}; unify((int -> int -> int = 'b -> 'c) {(int -> 'd) / 'c})
=
{('d -> 'e) / 'a}; {(int -> 'd) / 'c}; unify(int -> int -> int = 'b -> int -> 'd)
```

函数约束分解为两个较小的约束：

```text
...
=
{('d -> 'e) / 'a}; {(int -> 'd) / 'c}; unify(int = 'b, int -> int = int -> 'd)
```

我们得到另一个替换：

```text
...
=
{('d -> 'e) / 'a}; {(int -> 'd) / 'c}; {int / 'b}; unify((int -> int = int -> 'd) {int / 'b})
=
{('d -> 'e) / 'a}; {(int -> 'd) / 'c}; {int / 'b}; unify(int -> int = int -> 'd)
```

然后我们打破另一个函数约束：

```text
...
=
{('d -> 'e) / 'a}; {(int -> 'd) / 'c}; {int / 'b}; unify(int = int, int = 'd)
```

由此产生的第一个新约束是微不足道的，只是被删除：

```text
...
=
{('d -> 'e) / 'a}; {(int -> 'd) / 'c}; {int / 'b}; unify(int = 'd)
```

最后一个约束给了我们另一个替代：

```text
=
{('d -> 'e) / 'a}; {(int -> 'd) / 'c}; {int / 'b}; {int / 'd}
```

最后，我们将统一的替换输出应用于推断的类型
通过约束生成：

```text
('a -> 'b -> 'e) {('d -> 'e) / 'a}; {(int -> 'd) / 'c}; {int / 'b}; {int / 'd}
=
(('d -> 'e) -> 'b -> 'e) {(int -> 'd) / 'c}; {int / 'b}; {int / 'd}
=
(('d -> 'e) -> 'b -> 'e) {int / 'b}; {int / 'd}
=
(('d -> 'e) -> int -> 'e) {int / 'd}
=
(int -> 'e) -> int -> 'e
```

事实上，这与 OCaml 为原始数据推断出的类型相同
表达式：

```ocaml
# fun f -> fun x -> f (( + ) x 1);;
- : (int -> 'a) -> int -> 'a = <fun>
```

除了 OCaml 使用不同的类型变量标识符之外。 OCaml 对我们很好
并将类型变量“降低”为字母表中较小的字母。我们
只需做一些额外的工作也可以做到这一点。

**类型错误。** 实际上，还有另一部分需要类型推断。如果
统一失败，编译器或解释器需要产生有用的错误
消息。这是一个重要的工程挑战，我们在此不予讨论。
它需要跟踪的不仅仅是约束：我们需要知道为什么
引入了约束，以及违反约束的后果。我们还需要
跟踪约束回到产生它的词汇代码段，所以
程序员可以看到问题出现在哪里。既然有可能
约束可以以多种不同的顺序进行处理，有多种可能
可能产生的错误消息。弄清楚哪一个将领导
程序员找出错误的根本原因，而不是某些下游后果
其中，是一个正在进行的研究领域。

## Let 多态

现在我们将向我们的小语言添加 `let` 表达式：

```text
e ::= x | i | b | e1 bop e2
    | if e1 then e2 else e3
    | fun x -> e
    | e1 e2
    | let x = e1 in e2   (* new *)
```

{{ video_embed | replace("%%VID%%", "tB8sDHFT54I")}}

事实证明，它们的类型推断比可能的要棘手得多
预计。最简单的方法是添加此约束生成规则：

```text
env |- let x = e1 in e2 : t2 -| C1, C2
  if env |- e1 : t1 -| C1
  and env, x : t1 |- e2 : t2 -| C2
```

从类型检查的角度来看，这与我们一直使用的规则相同。
对于许多 `let` 表达式来说，它工作得很好。例如：

```text
{} |- let x = 42 in x : int -| {}
  {} |- 42 : int -| {}
  x : int |- x : int -| {}
```

问题是，当绑定的值是多态函数时，
规则生成的约束过于严格。例如，考虑
恒等函数：

```ocaml
let id = fun x -> x in
let a = id 0 in
id true
```

OCaml 可以轻松地将 `id` 的类型推断为 `'a -> 'a` 并允许它
适用于 `int` 和 `bool`。但上面的规则并非如此
允许对这两种类型的应用。当我们使用它时，我们会生成
以下类型和约束：

```text
{} |- let id = fun x -> x in (let a = id 0 in id true) : 'c -| 'a -> 'a = int -> 'b, 'a -> 'a = bool -> 'c
  {} |- fun x -> x : 'a -| {}
    x : 'a |- x : 'a -| {}
  id : 'a -> 'a |- let a = id 0 in id true : 'c -| 'a -> 'a = int -> 'b, 'a -> 'a = bool -> 'c   <--- POINT 1
    id : 'a -> 'a |- id 0 : 'b -| 'a -> 'a = int -> 'b
      id : 'a -> 'a |- id : 'a -> 'a -| {}
      id : 'a -> 'a |- 0 : int -| {}
    id : 'a -> 'a, a : 'b |- id true : 'c -| 'a -> 'a = bool -> 'c   <--- POINT 2
      id : 'a -> 'a, a : 'b |- id : 'a -> 'a -| {}
      id : 'a -> 'a, a : 'b |- true : bool -| {}
```

请注意，我们确实为 `id` 推断出类型 `'a -> 'a`，你可以在
示例后面几行的环境。但是，在第 1 点，我们推断
约束 `'a -> 'a = int -> 'b`，在第 2 点，我们推断
`'a -> 'a = bool -> 'c`。当统一算法遇到那些
约束，它将把它们分解为 `'a = int`, '`a = 'b`, `'a = bool`,
和`'a = 'c`。第一个和第三个是矛盾的，因为我们不能
有 `'a = int` 和 `'a = bool`。一个或另一个将被替换掉
在统一期间，留下不可满足的约束`int = bool`。那时
点统一将会失败，表明该程序类型错误。

问题是 `id` 的推断类型中的 `'a` 类型变量代表
对于未知但**固定**类型。在每次应用 `id` 时，我们想让
`'a` 成为 **不同** 类型，而不是强制它始终相同
类型。

{{ video_embed | replace("%%VID%%", "me-Ll7mjNh8")}}

`let` 表达式的多态性问题的解决方案并不简单。
它要求我们引入一种新的类型：*类型方案*。类型方案
类似于数理逻辑中的“通用量化”。例如，在
你可能会写出这样的逻辑：“对于所有自然数 $x$，它认为 $0 \cdot x
= 0$”。 “为所有人”是普遍的量化：它从一个
特别是 $x$ 并陈述了一个适用于所有自然数的属性。

类型方案写作 `'a . t`，其中 `'a` 是类型变量，`t` 是
可能出现 `'a` 的类型。例如，`'a . 'a -> 'a` 是一个类型方案。它
是接受 `'a` 类型的值并返回值的函数的类型
类型为 `'a`，对于所有 `'a`。因此，它是多态同一性的类型
函数。

在类型方案中，我们还可以在点的左侧放置许多类型变量。
例如， `'a 'b . 'a -> 'b -> 'a` 是一个函数的类型，它接受
两个参数并返回第一个。在 OCaml 中，我们可以将其写为
`fun x y -> x`。请注意，utop 正如我们所期望的那样推断出它的类型：

```ocaml
# let f = fun x y -> x;;
val f : 'a -> 'b -> 'a = <fun>
```

但我们实际上可以用类型方案手动写下注释：

```ocaml
# let f : 'a 'b . 'a -> 'b -> 'a = fun x y -> x;;
val f : 'a -> 'b -> 'a = <fun>
```

请注意，OCaml 接受我们的手动类型注释，但不包括
`'a 'b .` 其输出中的一部分。 **但它隐含地存在并且始终存在
** 一般来说，只要 OCaml 推断出类型 `t` 并且该类型已经具有
在其中输入变量，实际上它是一个类型方案。例如，类型
`List.length` 实际上是一个类型方案：

```ocaml
# let mylen : 'a . 'a list -> int = List.length;;
val mylen : 'a list -> int = <fun>
```

OCaml 只是不费心输出类型变量的列表
类型方案中点的左侧。实际上，它们只会使输出变得混乱，并且
许多程序员永远不需要了解它们。但现在你正在学习
类型推断，是时候让你知道了。

现在我们有了类型方案，我们将拥有将名称映射到的静态环境
类型方案。我们可以将类型视为类型方案的特殊情况
其中类型变量列表为空。对于类型方案，`let` 规则
与上面的天真的规则相比，仅以一种方式进行更改，即 `generalize` on
最后一行：

```text
env |- let x = e1 in e2 : t2 -| C1, C2
  if env |- e1 : t1 -| C1
  and generalize(C1, env, x : t1) |- e2 : t2 -| C2
```

`generalize` 的工作是采用像 `'a -> 'a` 这样的类型并_概括_它
在环境 `env` 中进入类似 `'a . 'a -> 'a` 的类型方案
约束`C1`。让我们稍后再回顾一下它是如何工作的。在此之前，
还有另一条规则需要更改，即名称规则：

```text
env |- n : instantiate(env(n)) -| {}
```

唯一改变的是 `instantiate` 的使用。它的工作是
采用像 `'a . 'a -> 'a` 这样的类型方案并将其实例化为新类型
（这里我们严格指的是类型，而不是类型方案）具有新的类型变量。
例如，如果 `'b` 不是，则 `'a . 'a -> 'a` 可以实例化为 `'b -> 'b`
但在其他地方用作类型变量。

以下是这两个修订后的规则如何协同工作以获得我们之前的示例
正确的识别函数：

```text
{} |- let id = fun x -> x in (let a = id 0 in id true)
  {} |- fun x -> x : 'a -> 'a -| {}
    x : 'a |- x : 'a -| {}
  id : 'a . 'a -> 'a |- let a = id 0 in id true   <--- POINT 1
```

让我们在第 1 点暂停。当 `id` 被 `let` 放入环境中时
规则，其类型从 `'a -> 'a` 泛化到 `'a . 'a -> 'a`；也就是说，从
类型到类型方案。这记录了 `id` 的每个应用程序的事实
应该为 `'a` 使用它自己的值。正在进行：

```text
{} |- let id = fun x -> x in (let a = id 0 in id true)
  {} |- fun x -> x : 'a -> 'a -| {}
    x : 'a |- x : 'a -| {}
  id : 'a . 'a -> 'a |- let a = id 0 in id true   <--- POINT 1
    id : 'a . 'a -> 'a |- id 0
      id : 'a . 'a -> 'a |- id : 'b -> 'b -| {}   <--- POINT 3
```

在第 3 点暂停，当 `id` 应用于 `0` 时，我们实例化其类型
变量 `'a` 与新类型变量 `'b`。让我们完成：

```text
{} |- let id = fun x -> x in (let a = id 0 in id true) : 'e -| 'b -> 'b = int -> 'c, 'd -> 'd = bool -> 'e
  {} |- fun x -> x : 'a -> 'a -| {}
    x : 'a |- x : 'a -| {}
  id : 'a . 'a -> 'a |- let a = id 0 in id true : 'e -| 'b -> 'b = int -> 'c, 'd -> 'd = bool -> 'e   <--- POINT 1
    id : 'a . 'a -> 'a |- id 0 : 'c -| 'b -> 'b = int -> 'c
      id : 'a . 'a -> 'a |- id : 'b -> 'b -| {}   <--- POINT 3
      id : 'a . 'a -> 'a |- 0 : int -| {}
    id : 'a . 'a -> 'a, a : 'b |- id true : 'e -| 'd -> 'd = bool -> 'e   <--- POINT 2
      id : 'a . 'a -> 'a, a : 'b |- id : 'd -> 'd -| {}   <--- POINT 4
      id : 'a . 'a -> 'a, a : 'b |- true : bool -| {}
```

在第 4 点，当 `id` 应用于 `true` 时，我们再次实例化其类型
变量 `'a` 具有新的类型变量，这次是 `'d`。所以约束条件
第 1 点和第 2 点收集的数据不再矛盾，因为它们是
谈论不同类型的变量。这些限制是：

```text
'b -> 'b = int -> 'c
'd -> 'd = bool -> 'e
```

因此，统一算法将得出结论：

```text
'b = int
'c = int
'd = bool
'e = bool
```

因此整个表达式被成功推断为类型 `bool`。

**实例化和泛化。**我们使用了两个新函数，`instantiate`
和 `generalize`，定义 `let` 表达式的类型推断。我们需要
定义这些函数。

最简单的是`instantiate`。  给定一个类型方案 `'a1 'a2 ... 'an . t`，
我们通过以下方式实例化它：

- 选择 `n` 新鲜类型变量，以及
- 将其中的每一个替换为 `t` 中的 `'a1` 到 `'an` 。

与评估中的情况相比，这里的替换并不复杂
替换模型，因为类型中没有任何内容可以绑定
变量名称。

但 `generalize` 需要更多工作。  这是 `let` 规则：

```text
env |- let x = e1 in e2 : t2 -| C1, C2
  if env |- e1 : t1 -| C1
  and generalize (C1, env, x : t1) |- e2 : t2 -| C2
```

为了概括 `t1`，我们执行以下操作。

首先，我们假装 `e1` 才是最重要的，而 `let` 的其余部分
表达式不存在。如果 `e1` 是整个程序，我们将如何完成
类型推断？我们在 `C1` 上运行统一算法，得到替换
`S`，并返回 `t1 S` 作为 `e1` 的推断类型。所以，现在就这样做。我们打电话吧
推断出类型 `u1`。我们还将 `S` 应用于 `env` 以获取新环境
`env1`，现在反映了我们从 `e1` 收集的所有类型信息。

其次，我们找出 `u1` 中的哪些类型变量应该被泛化。为什么
不是全部吗？因为某些类型变量可以通过代码引入
围绕 `let` 表达式，例如，

```ocaml
fun x ->
  (let y = e1 in e2) (let z = e3 in e4)
```

`x` 的类型变量不应在推断类型时进行泛化
`y` 或 `z`，因为 `x` 必须在所有四个中具有相同的类型
子表达式，`e1` 到 `e4`。泛化可能会错误地允许 `x`
`e1` 和 `e2` 中有一种类型，但 `e3` 和 `e4` 中有不同的类型。

因此，我们只概括 `u1` 中**存在**但在 `u1` 中**不**的变量
`env1`。这样我们只概括 `e1` 中的类型变量，而不是变量
当我们开始推断 `let` 时，它们已经在环境中了
表达式的类型。假设这些变量是 `'a1 ... 'an`。我们的类型方案
给 `x` 则为 `'a1 ... 'an . u1`。

将所有这些放在一起，我们最终得到：

```text
generalize(C1, env, x : t1) =
  env1, x : 'a1 ... 'an . u1
```

回到上面的识别函数的例子，我们有
`generalize({}, {}, x : 'a -> 'a)`。在这个相当简单的情况下，`unify`
从环境中没有发现新的平等，因此 `u1 = 'a -> 'a` 和
`env1 = {}`。 `u1` 中唯一的类型变量是 `'a`，并且它没有出现在
`env1`。因此 `'a` 被泛化，产生 `'a . 'a -> 'a` 作为类型方案
`id`。

## 多态性和可变性

{{ video_embed | replace("%%VID%%", "6tj9WrRqPeU")}}

`let` 表达式的类型推断还有一个复杂之处。它
当我们向语言添加可变引用时出现。考虑这个例子
代码，在 OCaml 中不进行类型检查：

```ocaml
let succ = fun x -> ( + ) 1 x;;
let id = fun x -> x;;
let r = ref id;;
r := succ;;
!r true;;  (* error *)
```

很明显，我们应该推断 `succ : int -> int` 和 `id : 'a . 'a -> 'a`。但是
`r` 的类型应该是什么？我们很容易说我们应该推断
`r : 'a . ('a -> 'a) ref`。这将使我们实例化 `r` 的类型为
第 4 行上的 `(int -> int) ref` 并将 `succ` 存储在 `r` 中。但这也让我们
在第 5 行将 `r` 的类型实例化为 `(bool -> bool) ref`。
灾难：它会导致将 `succ` 应用于 `true`，这不是类型安全的。

OCaml及相关语言采用的解决方案称为*value
限制：* 类型系统旨在防止多态可变值
不再持有不止一种类型。让我们再重做一下这个例子，
停下来看看顶层输出：

```ocaml
# let id = fun x -> x;;
val id : 'a -> 'a = <fun>   (* as expected *)

# let r = ref id;;
val r : ('_weak1 -> '_weak1) ref = { ... }   (* what is _weak? *)

# r;;
- : ('_weak1 -> '_weak1) ref = { ... }   (* it's consistent at least *)

# r := succ;;
- : unit = ()

# r;;
- : (int -> int) ref = { ... }   (* did r just change type ?! *)
```

当推断出 `r` 的类型时，OCaml 会为其提供一个涉及 _weak_ 类型的类型
变量。所有此类变量的名称均以 `'_weak` 开头。弱型
变量是一个尚未泛化的变量，因此无法实例化
多种类型。相反，它表示一种尚不为人所知的单一类型。思考
因为该变量的类型推断尚未完成：OCaml 正在等待
以获得更多信息以准确确定它是什么。当 `r := succ` 为
执行后，该信息最终变得可用。 OCaml 推断
`'_weak1 = int` 来自 `succ` 的类型。然后 OCaml 将 `'_weak1` 替换为
`int` 无处不在。这就是最后一行产生错误的原因：

```text
# !r true;;
Error: This expression has type bool but an expression was expected of type int
```

由于 `r : (int -> int) ref`，我们不能将 `!r` 应用于 `bool`。

这里我们不会介绍弱类型变量的实现。

但是，我们不要离开多态类型和多态类型之间的相互作用这个话题。
可变性还没有。你可能会认为这是一种现象
仅影响 OCaml。但事实上，即使是 Java 也受到了影响。

考虑以下类层次结构：

```java
class Animal { }
class Elephant extends Animal { }
class Rabbit extends Animal { }
```

现在假设我们创建了一系列动物：

```java
Animal[] a= new Rabbit[2]
```

这里我们使用*子类型多态性*来分配 `Rabbit` 对象的数组
到 `Animal[]` 参考。这与*参数多态性*不同
我们一直在 OCaml 中使用，但它仍然是多态性。

如果我们尝试一下会怎样？

```java
a[0]= new Elephant()
```

由于 `a` 被输入为 `Animal` 数组，因此我们可以
向其中分配一个大象对象，就像我们可以分配一个兔子对象一样。并且
事实上，根据 Java 编译器的判断，该代码没有问题。但是Java给了我们一个
如果我们运行该代码，则会出现运行时错误！

```java
Exception java.lang.ArrayStoreException
```

问题是，将第一个数组元素突变为兔子会留下
我们使用一个 `Rabbit` 数组，其中一个元素是 `Elephant`。 （哎呀！安
大象会坐在兔子身上。可怜的小面包。）但是在 Java 中，每个类型
数组的对象应该是整个数组的属性。每个
因此，由 `new Rabbit[2]` 创建的数组的元素必须是 `Rabbit`。所以
Java 通过在运行时检测错误来防止上述分配
引发异常。

这实在是另一种形式的价值限制！值的类型
根据值限制，存储在可变位置中的值可能不会更改。
对于数组，Java 通过运行时检查来实现，而不是拒绝
程序在编译时。这在稳健性之间取得了平衡
（防止错误发生）和表现力（允许更多无错误
程序进行类型检查）。
