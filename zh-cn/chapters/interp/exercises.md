{{ ex4 | replace("%%NAME%%", "generalize patterns")}}

在Core OCaml中，只有两种模式：`Left x`和`Right x`，
其中 `x` 是变量名。  但在完整的 OCaml 中，模式远不止这些
一般。让我们看看我们可以在 Core OCaml 中概括模式到什么程度。

**第 1 步：** 这是模式的 BNF 语法，并稍作修改
表达式的 BNF 语法：

```text
p ::= i | (p1, p2) | Left p | Right p | x | _

e ::= ...
    | match e with | p1 -> e1 | p2 -> e2 | ... | pn -> en
```

在 `match` 的修订语法中，只有第一个 `|` 位于该行，
紧接在关键字 `match` 之前的是元语法。  剩余的
该行的四个 `|` 是语法。  请注意，我们在之前需要 `|`
第一个模式。

**步骤 2：** 如果通过替换任何值，则值 `v` 与模式 `p` 匹配
`p` 中的变量或通配符带有值，我们可以准确地获得 `v`。
例如：

* `2` 与 `x` 匹配，因为 `x{2/x}` 是 `2`。
* `Right(0,Left 0)` 匹配 `Right(x,_)` 因为
`Right(x,_){0/x}{Left 0/_}` 是 `Right(0,Left 0)`。

让我们定义一个名为 `matches` 的新三元关系，以这些关系为指导
例子：

```text
v =~ p // s
```

将此关系发音为“`v` 匹配 `p` 产生替换 `s`”。

这里，`s`是一系列替换，例如`{0/x}{Left 3/y}{(1,2)/z}`。
这种关系只有一条规则：

```text
v =~ p // s
  if v = p s
```

例如，

```text
2 =~ x // {2/x}
  because 2 = x{2/x}
```

**步骤 3：** 计算匹配表达式：

* 评估与值匹配的表达式。
* 如果该表达式与第一个模式匹配，则计算该表达式
对应于该模式。
* 否则，匹配第二个模式、第三个模式等。
* 如果没有任何模式匹配，则评估被“卡住”：它不能采取任何模式
更多步骤。

利用这些见解，完成以下评估规则
填写标有 `???` 的位置：

```text
(* This rule should implement evaluation of e. *)
match e with | p1 -> e1 | p2 -> e2 | ... | pn -> en
--> ???
  if ???

(* This rule implements moving past p1 to the next pattern. *)
match v with | p1 -> e1 | p2 -> e2 | ... | pn -> en
--> match v with | p2 -> e2 | ... | pn -> en
  if there does not exist an s such that ???

(* This rule implements matching v with p1 then proceeding to evaluate e1. *)
match v with | p1 -> e1 | p2 -> e2 | ... | pn -> en
--> ??? (* something involving e1 *)
  if ???
```

请注意，我们不需要显式地编写以下规则：

```text
match v with |  -/->
```

评估将陷入困境，因为其他三个都没有
上述规则将适用。

**第 4 步：** 通过评估以下内容来仔细检查你的规则
表达式：

`match (1 + 2, 3) with | (1,0) -> 4 | (1,x) -> x | (x,y) -> x + y`

{{ ex4 | replace("%%NAME%%", "let rec")}}

`let` 的评估规则之一是

```text
let x = v in e --> e{v/x}
```

我们可以尝试将其调整为 `let rec`：

```text
let rec x = v in e --> e{v/x}   (* broken *)
```

但该规则无法正常工作，如以下示例所示：

```text
  let rec fact = fun x ->
    if x <= 1 then 1 else x * (fact (x - 1)) in
  fact 3

-->

  (fun x -> if x <= 1 then 1 else x * (fact (x - 1)) 3

-->

  if 3 <= 1 then 1 else 3 * (fact (3 - 1))

-->

  3 * (fact (3 - 1))

-->

  3 * (fact 2)

-/->
```

我们现在陷入困境，因为我们需要评估 `fact`，但它不执行。在
本质上，我们使用的语义规则“忘记”了应该具有的函数值
与 `fact` 相关联。

解决这个问题的一个好方法是引入一种新的语言结构
递归简单地称为 `rec`。 （请注意，OCaml 没有任何构造
直接对应于 `rec`。）形式上，我们扩展了表达式的语法
如下：

```text
e ::= ...
    | rec f -> e
```

并添加以下评价规则：

```text
rec f -> e  -->  e{(rec f -> e)/f}
```

这条规则的直观解读是，在评估时
`rec f -> e`，我们在`e`的主体中“展开”`f`。  例如，
这是一个用 `rec` 编码的无限循环：

```text
  rec f -> f

-->  (* step rec *)

  f{(rec f -> f)/f}

= (* substitute *)

  rec f -> f

--> (* step rec *)

  f{(rec f -> f)/f}

...
```

现在我们可以使用`rec`来实现`let rec`。 `let rec` 出现在
节目：

```ocaml
let rec f = e1 in e2
```

我们*去糖*（即重写）它

```ocaml
let f = rec f -> e1 in e2
```

请注意，第二次出现的 `f`（在 `rec` 内部）会遮盖第一个出现的情况。
回到 `fact` 示例，它的脱糖版本是

```ocaml
let fact = rec fact -> fun x ->
  if x <= 1 then 1 else x * (fact (x - 1)) in
fact 3
```

评估以下表达式（我们认为有 17 个步骤，尽管它确实变得很漂亮
乏味）。你可能想通过写“F”来简化你的生活
`(rec fact -> fun x -> if x <= 1 then 1 else x * (fact (x-1)))`

```text
let rec fact = fun x ->
  if x <= 1 then 1 else x * (fact (x - 1)) in
fact 3
```

使用以下（避免捕获）替换规则，类似于
`let` 和 `fun` 的规则：

```text
(rec f -> e){v/x} = rec f -> e{v/x}
  if x <> f
  and f not in FV(v)

(rec f -> e){v/f} = rec f -> e
```
