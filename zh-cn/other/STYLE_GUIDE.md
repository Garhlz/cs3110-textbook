# CS3110 教科书中文翻译风格指南

**版本**：1.0  
**日期**：2026-04-22  
**适用范围**：`zh-cn/chapters/**/*.md` 所有文件

---

## 📖 总体原则

1. **忠实原意**：不新增、不删减、不改变技术事实和教学逻辑
2. **自然流畅**：避免生硬逐字翻译，符合中文教材语气
3. **术语统一**：全书同一概念使用同一术语
4. **结构保护**：保留所有 Markdown/MyST 特殊元素
5. **混合使用**：适当保留英文术语，避免不必要的翻译

## 🔤 中英混用规范

### 规则 1: 保留英文（必须）

永远**保持英文**的内容：

```markdown
// ✅ 保留英文

- OCaml, OCaml-Jupyter, Jupyter Book, MyST Markdown, Jupytext
- 代码中出现的：utop, opam, List.map, Array.length, Hashtbl
- 数据类型名：int, float, string, bool, array, ref
- 模块名：List, Array, Hashtbl, Printf
- 函数名：fold, map, filter, reduce, zip
- 关键字：let, rec, if, then, else, match, with, fun, function
- 行内代码：`List.map`, `fun x -> x + 1`, `let x = 5`
- 变量名和参数名：在文中提到时用反引号，e.g. `lst`, `elt`
- 技术对象在英文原文中的拼写：cons, head, tail, nil
```

### 规则 2: 优先翻译（必须）

以下概念应该**翻译为中文**：

```markdown
// ✅ 翻译为中文

- 普通名词：列表、元组、记录、变体、选项、异常、函数、方法
- 概念：模式匹配、递归、类型推断、类型注解、值、表达式
- 动作：定义、定义、计算、评估、推断、检查、构建、实现
- 限定词：递归的、不可变的、可变的、一等的、原始的
- 说明性文本：所有段落、列表项、表格、图注
```

### 规则 3: 首次出现标注（推荐）

对于**重要但可能陌生的概念**，首次出现时可以标注：

```markdown
// ✅ 推荐格式

这个特性称为 functor（函子）...（后文直接用 functor）
我们使用 monad（单子）来...（后文直接用 monad）
cons 运算符（来自 Lisp 的 "construct"）...
```

### 规则 4: 上下文判断

根据**上下文**决定保留或翻译：

```markdown
// 例子 1: 函数名 vs 动作
在函数定义中：let map = ...             // map 保留，代码语境
在说明中：我们用 map 函数来...          // map 保留
在说明中：为了映射这个列表...           // "映射"可翻译，通俗语境

// 例子 2: 数据类型 vs 通俗说法
在代码中：[1; 2; 3] : int list          // int list 保留
在说明中：整数列表                       // "整数列表"翻译

// 例子 3: 术语 vs 描述
术语化：cons 运算符                      // 术语，保留英文
描述：构造列表头部                       // 描述，翻译
```

## 🗣️ 表达和句式风格

### 规则 5: 避免逐字翻译

| 原文 | ❌ 逐字翻译 | ✅ 自然翻译 | 说明 |
|------|----------|----------|------|
| before we dive in | 潜水之前 | 在深入学习之前 | idiom 需要转换 |
| all of which | 其中许多人 | 其中许多 | 删除不必要的词 |
| There's a lot of similarity | 有很多相似之处 | 这两个函数有许多相似之处 | 前后文连贯 |

### 规则 6: 主语和连接词

中文教材常用"我们"作为通用主语，但要**保持一致**：

```markdown
// ✅ 一致使用第二人称或"我们"

版本A（第二人称）：
你会看到这个列表包含...
当你定义一个函数时...
你需要理解这个概念...

版本B（"我们"）：
我们会看到这个列表包含...
当我们定义一个函数时...
我们需要理解这个概念...

// ❌ 不要混用
你会看到...我们定义...（混用不当）
```

**本书建议**：优先使用**"我们"**（包括讲者和读者的虚拟第一人称）。

### 规则 7: 列表和连接词

#### 列表连接

```markdown
// ✅ 推荐

这个概念有三个方面：

- 首先，...
- 其次，...
- 最后，...

或者：

- 语法：...
- 语义：...
- 工具：...
```

#### "和"的使用

```markdown
// ✅ 推荐

我们学习列表、元组和记录这三种数据结构。
（多个平等项目用顿号分隔，最后一个用"和"）

当列表为空时，我们返回 nil，否则我们...
（复句用"，"或"；"连接，明确逻辑关系）
```

### 规则 8: 代码说明的位置和语气

```markdown
// ✅ 推荐：说明在代码前或代码后

代码**前**（指示意图）：
我们定义一个函数来计算列表长度：
```ocaml
let rec length = function
  | [] -> 0
  | _ :: t -> 1 + length t
```

代码**后**（说明结果）：
上面的函数递归地遍历列表，每次碰到一个元素就计数加一。

// ✅ 推荐：同一段落内保持指代一致

函数 `add1` 接收一个列表，对其中每个元素加一。
（后续不要改成"该列表被传入..."）

// ❌ 避免

函数 `add1` 接收一个列表。其中每个元素都会被加一。
（改变了主语和语态）
```

### 规则 9: 否定和条件句

```markdown
// ✅ 推荐

如果列表不为空...
当元素不存在时...
即使没有安装...

// 偶尔可用，但要谨慎

除非你已经了解...
不需要安装...

// ❌ 过于复杂

如果你不没有理解...（双否定）
```

## 📚 术语使用表

### 核心术语（必须统一）

| 概念 | 中文翻译 | 英文保留 | 首次标注 | 例句 |
|------|--------|--------|--------|------|
| cons operator | cons 运算符 | cons :: | 可选 | 使用 cons 运算符将元素添加到列表 |
| head/tail | 头部/尾部 或 保留 | head, tail | 可选 | 列表的头部是第一个元素... |
| pattern matching | 模式匹配 | - | 否 | 我们使用模式匹配来分解... |
| list | 列表 | list | 否 | OCaml 列表是单链表... |
| function | 函数 | function | 否 | 定义一个函数需要... |
| type inference | 类型推断 | - | 否 | OCaml 使用类型推断来... |
| higher-order function | 高阶函数 | - | 否 | 高阶函数接收其他函数作为参数... |

### 容易混淆的术语对

| 英文 | 中文选项 | 本书选择 | 原因 |
|------|--------|--------|------|
| argument vs parameter | 实参/形参 vs 参数 | 统一用"参数"，必要时加"实参/形参"的前缀 | 原文不严格区分 |
| lambda vs anonymous function | lambda / λ vs 匿名函数 | 在代码语境保留 `fun x -> ...`，说明时用"匿名函数"或"lambda 函数" | 教学清晰 |
| functor vs function | 函子（来自 Lisp/范畴论）vs 函数 | functor 保留英文，首次标注"函子"；function 译为"函数" | OCaml 术语精准 |
| value vs variable | 值 vs 变量 | 区分：不可变的是值，可变的是变量 | 符合函数式编程哲学 |

## ✏️ 常见修改示例

### 例 1: 逐字翻译的修改

```markdown
// 原文
before we dive in to learning OCaml

// ❌ 初稿（逐字翻译）
潜水之前深入学习 OCaml

// ✅ 润色
在我们深入学习 OCaml 之前

// 或

让我们开始学习 OCaml
```

### 例 2: 列表连接的修改

```markdown
// 原文
- lists and tuples, might feel similar to Python; and
- records and variants, might feel similar to struct and enum types from C or Java.

// ❌ 初稿
- **列表**和**元组**，可能感觉与Python类似；和
- **记录**和**变体**，可能感觉类似于 `struct` 和 `enum` 类型
来自 C 或 Java。

// ✅ 润色（选项 1）
- **列表**和**元组**可能与 Python 中的类似；
- **记录**和**变体**可能与 C 或 Java 中的 `struct` 和 `enum` 类型相似。

// ✅ 润色（选项 2，更简洁）
我们会学习：
- **列表**和**元组**（类似 Python）
- **记录**和**变体**（类似 C/Java 中的 struct/enum）
```

### 例 3: 完全错译的修改

```markdown
// 原文
Building Lists

// ❌ 初稿
建筑清单

// ✅ 润色
构建列表

---

// 原文
The double-colon operator is pronounced "cons"

// ❌ 初稿
双冒号运算符发音为"缺点"

// ✅ 润色
双冒号运算符发音为"cons"（来自 Lisp 的"构造"）
```

### 例 4: 上下文连贯的修改

```markdown
// 原文
There's a lot of similarity between those two functions:
- They both pattern match against a list.
- They both return the same value for the base case of the empty list.

// ❌ 初稿
这两个函数有很多相似之处：
- 它们都针对列表进行模式匹配。
- 对于空列表的基本情况，它们都返回相同的值。

// ✅ 润色
这两个函数有很多相似之处：
- 都对列表进行模式匹配；
- 在基本情况（空列表）下返回相同的值；
- 在非空情况下都对尾部递归调用。
```

## 🚫 常见错误避免

| 错误 | 原因 | 如何避免 |
|------|------|--------|
| "潜水" 代替"深入" | 逐字翻译 idiom | 先查英文短语，理解深层意思 |
| "缺点" 代替"cons" | 机器翻译搞错 | 理解技术术语背景（Lisp origin） |
| "其中许多人" | 过度翻译"all of which" | 根据语境判断，通常不需要"人"字 |
| 混用"head"和"头部" | 术语不统一 | 制定章节级别的术语决策 |
| "来自 C 或 Java。" 单独成行 | 分行不当 | 检查句式是否完整，必要时调整段落 |
| "build lists" → "建筑清单" | 机器翻译歧义 | "建造"vs"构建"需要上下文判断 |

## 💡 推荐润色步骤

对于每个文件，按以下顺序修改：

1. **第一遍**：搜索"坏词"列表，快速替换明显的错误
2. **第二遍**：对比原文和初稿，逐段审查
3. **第三遍**：检查术语一致性（especially head/tail, cons, pattern matching）
4. **第四遍**：检查格式（列表、链接、代码块）
5. **第五遍**：通读全文，确保连贯性
